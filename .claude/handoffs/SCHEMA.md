# ハンドオフ プロトコル仕様

> エージェント間でタスクを引き継ぐ標準形式。
> 人間が介在せずにエージェントAのアウトプットをエージェントBが受け取れる。

---

## ファイル命名規則

```
.claude/handoffs/active/[YYYYMMDD-HHMM]_[from]_[to]_[task-slug].md
.claude/handoffs/completed/[同上].md  ← 完了後に移動
```

例:
```
.claude/handoffs/active/20260325-1430_strategy-lead_proposal-writer_saas-lp.md
```

---

## ハンドオフファイル フォーマット

```markdown
---
id: [YYYYMMDD-HHMM]_[from]_[to]
from: [エージェント名]
to: [エージェント名]
status: pending | in-progress | completed | rejected
priority: high | medium | low
created: YYYY-MM-DDTHH:MM:SS
project: .claude/projects/[プロジェクトファイル名].md
---

## タスク（1文）
[受け取るエージェントが何をするか — 動詞で始める]

## なぜこのタスクが必要か（背景）
[前工程で何が明らかになり、なぜ次が必要か — 2〜3文]

## インプット
[実際のデータ・分析結果・ファイルパス。「市場データ」ではなく数値・ソース付き]

## 期待アウトプット
- 形式: [Markdown / PowerPoint / コード / その他]
- 分量: [ページ数 / 文字数 / 概算]
- 期限: [即日 / ○日以内 / ○月○日]

## 成功基準
[何ができていれば「完了」か — チェックボックス形式]
- [ ]
- [ ]

## 参照スキル
- [スキルファイル名] §N [セクション名]

## 制約
- [守るべきルール / 使ってはいけない手法 / 禁止表現]

## 反証チェック結果（送り出し側）
- ✅ Step 1（自己反証）:
- ✅ Step 2（構造反証）:
- ✅ Step 3（実用反証）:
- 🔺 残存リスク:
```

---

## ステータス管理

| ステータス | 意味 | 次のアクション |
|---|---|---|
| `pending` | 受け取り待ち | 受け取り側エージェントが着手前に確認 |
| `in-progress` | 作業中 | 受け取り側が着手時に更新 |
| `completed` | 完了 | `completed/` ディレクトリに移動 |
| `rejected` | 差し戻し | 送り出し側が修正して再送 |

---

## 受け取り拒否（差し戻し）手順

以下のいずれかに該当する場合、即座に `status: rejected` に変更して理由を記載:

```markdown
## 差し戻し理由
- [ ] 数値なしのPL依頼（根拠数値が不足）
- [ ] ソースなしの情報（未検証データのみ）
- [ ] 反証チェック未実施（前工程のStep 1〜3が空）
- [ ] 成功基準が不明確（「いい感じに」等の抽象指示）
```

---

## Stop Hook 検知仕様

セッション終了時（Stop hook）に `.claude/handoffs/active/` を自動スキャン。
未処理ハンドオフがある場合は警告を出力:

```
⚠️  未処理ハンドオフが N 件あります:
  → [ファイル名]: [from] → [to] | [タスク1文]
次のセッション開始時に to エージェントを起動してください。
```
