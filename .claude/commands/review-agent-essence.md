# /review-agent-essence — エージェント本質レビュー

引数: $ARGUMENTS（エージェント名 or "all"）

## 目的
エージェントの設計自体を「本質」の視点でレビューし、矛盾・形骸化・過剰設計を検出する。
`/evolve`がスキルの品質改善なら、`/review-agent-essence`はエージェント設計の本質レビュー。

## レビュー手順

### Phase 1: 本質チェック（このエージェントは何者か）
1. 思想的基盤の人物は、このエージェントの判断に本当に影響しているか？
2. シナリオプレイブックで、思想が具体的な判断ステップに接続されているか？
3. 干渉原則（佐藤×小野寺）がこのエージェントの文脈で意味を持っているか？
4. 「このエージェントがいなかったら何が困るか」を1文で言えるか？言えなければ存在意義が曖昧

### Phase 2: 矛盾チェック
1. 思想的基盤と禁止事項が矛盾していないか
2. 参照スキルの内容とシナリオプレイブックが整合しているか
3. 連携先エージェントとの責務境界が明確か（重複・空白がないか）
4. model:設定（opus/sonnet）がこのエージェントの判断の重さに合っているか

### Phase 3: 実践知チェック
1. シナリオは実務で本当に使えるか？（机上の空論になっていないか）
2. 判断分岐があるか？（「1.調査 2.分析 3.報告」の汎用フローになっていないか）
3. 禁止事項は具体的か？（「品質を保つ」のような曖昧な禁止になっていないか）
4. このエージェントのアウトプットで、クライアントの行動が変わるか？

### Phase 4: 濃度チェック
1. 行数に対して情報密度は十分か？（冗長な説明がないか）
2. 他エージェントと重複している記述はないか？
3. DRY原則に違反していないか（CLAUDE.mdの共通ルールと重複していないか）

## 出力フォーマット

```
# エージェント本質レビュー: [agent-name]

## 本質スコア（各1-5, 合計20点満点）
  E1 存在意義の明確さ:  [1-5]
  E2 思想の接続度:      [1-5]
  E3 実践知の深さ:      [1-5]
  E4 設計の無矛盾性:    [1-5]
  合計: [XX/20]

## 発見された問題
1. [問題]: [具体的な箇所と修正案]

## 推奨アクション
- [アクション1]
- [アクション2]
```

## 引数なしの場合
全34エージェントの本質スコアをランキング形式で出力し、最もスコアが低い3体の詳細レビューを実行する。
