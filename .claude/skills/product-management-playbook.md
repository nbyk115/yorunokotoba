# Product Management Playbook — プロダクト戦略・PMF検証・ロードマップ設計

> **参照エージェント**: product-manager, feedback-synthesizer, tech-lead, strategy-lead
> **用途**: PMF検証・機能優先順位付け・ロードマップ策定・プロダクトレビュー・GTM設計

---

## §1 PMF（Product-Market Fit）の定義と計測

### PMFとは何か
```
PMF = 「十分な大きさの市場」で「顧客の課題を本当に解決」し「再現性よく顧客獲得できる」状態。

PMFの誤解:
❌ 「ユーザーが増えればPMFした」 → 単なる成長はPMFではない
❌ 「ユーザーの声が良ければPMFした」 → 感覚的な満足はPMFではない
✅ PMFは「解約率・Retention・NPS・支払い意思額」で測定する
```

### PMF測定指標（優先順）

| 指標 | PMF基準値 | 測定方法 |
|---|---|---|
| **Day30 Retention** | 40%以上 | コホート分析（必須） |
| **Weekly Active Rate** | 40%以上（B2B）/ 20%以上（B2C） | MAU/DAU比 |
| **Sean Ellis Test** | 「非常に失望する」40%以上 | PMFサーベイ |
| **NPS** | 30以上 | 月次サーベイ |
| **Organic Growth** | 新規の20%以上が口コミ由来 | アトリビューション分析 |
| **MRRチャーン** | 月次2%以下 | ARR変動追跡 |

### PMF検証プロセス
```
Step 1: 仮説定義
  - ターゲット顧客セグメントを1つに絞る（「全員」は対象ゼロと同じ）
  - 解決する課題を1文で定義する
  - 成功の定義（数値）を事前に設定する

Step 2: MVP構築（最小検証単位）
  - 「これ1機能でPMFを測れるか」を先に問う
  - 開発前に「どのRetention数値が出れば次に進むか」を合意する

Step 3: 計測・判定
  - 2週間スプリント×最大6サイクル（3ヶ月）で判定する
  - PMF達成 / ピボット（課題 or セグメントを変える）/ 撤退 の3択で判断

Step 4: PMF後のスケール
  - PMF前にスケールしない（資金と時間の無駄）
  - PMF後に初めてマーケティング・セールス・採用を加速する
```

## §2 機能優先順位付け（RICE × PL）

### RICEスコア計算
```
RICE = (Reach × Impact × Confidence) ÷ Effort

Reach（到達ユーザー数/月）: 実数で入力
Impact（インパクト）: 0.25（最小）/ 0.5 / 1 / 2 / 3（最大）
Confidence（確信度）: 50% / 80% / 100%（データ根拠に基づく）
Effort（人週）: 実工数見積もり

例:
  機能A: 1000ユーザー × 2 × 80% ÷ 2週 = 800
  機能B: 500ユーザー × 3 × 50% ÷ 1週 = 750
  → 機能Aを優先
```

### PLインパクト連動（重要）
```
RICEスコアだけでは事業貢献が見えない。PLインパクトと組み合わせる。

PLインパクト試算（簡易）:
  チャーン改善機能 = チャーン改善率 × 顧客数 × ARPU
  例: -0.5%チャーン × 500社 × 5万円 = 125万円/月

獲得改善機能 = CVR改善率 × 月間MQL × ACV ÷ 12
  例: +1% CVR × 1000MQL × 240万ACV ÷ 12 = 20万円/月

→ PLインパクトが大きく、RICEスコアも高い機能を最優先とする
```

### バックログ管理の原則
- **P0（緊急）**: 既存顧客に影響するバグ・セキュリティ問題
- **P1（重要）**: PMF改善・主要チャーン防止・大型商談の前提条件
- **P2（通常）**: 新規獲得改善・UX改善・小さな追加機能
- **P3（後回し）**: あると便利な機能・内部ツール改善

## §3 ロードマップ設計

### 四半期ロードマップテンプレート
```
【FY20XX Q[X] ロードマップ】
テーマ: ___（例: チャーン改善 / 新セグメント開拓）

Must-have（PMF直結・必達）:
  - [ ] 機能A: [PLインパクト] [担当] [完了基準]
  - [ ] 機能B: [PLインパクト] [担当] [完了基準]

Should-have（重要だが条件付き）:
  - [ ] 機能C: [理由] [前提条件] [担当]

Nice-to-have（余力があれば）:
  - [ ] 機能D: [理由]

対象外（意図的にやらないこと）:
  - [やらない理由を必ず明示する]

成功の定義（Q末時点で何が達成されていればOKか）:
  - KPI1: [目標値]
  - KPI2: [目標値]
```

### ロードマップ変更の判断基準
```
変更すべき理由（正当）:
  - ユーザーデータが仮説を否定した
  - 競合が重要な機能をリリースした
  - 大型顧客の獲得・維持のために必要

変更してはいけない理由（不正当）:
  - 声の大きいユーザー1人の要望
  - 「なんとなく大切そう」な機能
  - エンジニアが作りたいもの
```

## §4 PRD（プロダクト要求仕様書）テンプレート

```markdown
# [機能名] PRD

## 概要
1文でこの機能が何をするかを説明する。

## 背景と課題
- なぜこの機能が必要か（ユーザーインサイト・データ根拠）
- この機能がなければ何が起きるか（放置コスト）

## ターゲットユーザー
- Who: [ユーザーセグメント]
- When: [どの文脈・場面で使うか]
- Job-to-be-Done: [ユーザーが「雇いたい」仕事]

## 成功指標（事前設定・必須）
- 主要KPI: [具体的な数値目標]
- 計測方法: [どのツール・クエリで測るか]
- 計測開始日: [リリース後X日後から]

## 機能仕様
### ユーザーストーリー
As [ユーザータイプ], I want [機能], so that [目的・価値].

### 受け入れ基準（Acceptance Criteria）
- [ ] [条件1]
- [ ] [条件2]

### 対象外（明示的にやらないこと）
- [除外事項1]

## 技術制約
- [tech-leadと事前合意した制約事項]

## リリース計画
- [ ] ベータ版（[対象顧客] / [規模]）
- [ ] 一般公開
```

## §5 スプリントレビュー・振り返り

### スプリントレビュー（2週間ごと）
```
□ 完了したUserStoryと受け入れ基準の達成確認
□ 今スプリントで学んだこと（仮説の検証結果）
□ 次スプリントの優先順位（変更があれば根拠を示す）
□ ブロッカーの共有と解決者の決定
```

### 四半期振り返り（PMF・戦略レビュー）
```
1. ロードマップ達成率と未達の理由
2. ユーザーRetentionとチャーンの変化（コホート分析）
3. 「やって良かった機能」と「やるべきでなかった機能」
4. 次四半期のテーマ（継続 / ピボット / 新展開）
5. 仮説リストの更新（検証済み / 棄却 / 新規追加）
```

## §6 プロダクト×営業×CSの連携設計

```
【プロダクト主導成長（PLG）モデル】

営業が関与しない成長:
  → フリートライアル設計（Activation率が命）
  → バイラル機能（招待・共有・埋め込み）
  → セルフサービスオンボーディング

営業が必要な成長（エンタープライズ）:
  → プロダクトで触れてから営業が関与する（PQL: Product Qualified Lead）
  → 「無料 → 有料」の転換シグナルをプロダクトで検知 → 営業に通知

CSが解決する問題をプロダクトで解決する:
  → CSにくる問い合わせTop10を分析 → プロダクトのUX改善で削減
  → ヘルススコアの定義をプロダクトの利用データから構築
```

## 🔺 反証モード（トリプルチェック必須）

本スキルを適用する全アウトプットは、CLAUDE.md「反証モード」セクションに定義されたトリプルチェックを必ず実行すること。

1. **自己反証**: 結論に対する反論・反例を最低3つ挙げ、確証バイアスを排除
2. **構造反証**: ロジックの飛躍・数値の妥当性・抜け漏れを検証
3. **実用反証**: 実行可能性・エッジケース・最悪シナリオを検証

チェックなしのアウトプットはドラフト扱い。省略禁止。

## §7 プロダクト組織設計とOKR連動

### プロダクトチーム構造の選択
| 構造 | 適合フェーズ | メリット | デメリット |
|---|---|---|---|
| 機能別チーム | 初期（5人未満） | 意思決定速度 | スケールしない |
| ミッション型チーム | 成長期（10-50人） | 自律性・所有感 | 調整コスト |
| プラットフォーム型 | スケール期（50人+） | 一貫性・効率 | 官僚化リスク |

### プロダクトOKR設計テンプレート
```
Objective: ユーザーが[具体的な成果]を達成できるようになる

KR1: [ユーザー行動指標]が XX → YY になる（確信度: X/5）
KR2: [ビジネス指標]が XX → YY になる（確信度: X/5）
KR3: [品質/満足度指標]が XX → YY になる（確信度: X/5）

PLインパクト: KR達成により月間粗利 +XX万円（kpi-analyticsで精緻化）
```

### プロダクト意思決定の黄金律
1. **データ優先、ただし全てをデータで決めない**: N=5のインタビューが統計より重要なこともある
2. **最速で学べる方法を選ぶ**: 完璧な機能より、仮説を検証できる最小の実装
3. **Type 1 / Type 2 判断**: 不可逆な決定（Type 1）は慎重に。可逆な決定（Type 2）は素早く
4. **ユーザーの言葉より行動データを信頼する**: 「欲しい」と「使う」は別物

## §8 競合製品分析（プロダクト視点）

### Jobs-to-be-Done 競合マップ
```
ユーザーが「雇う」製品・サービスは何か？

直接競合（同じジョブを同じ手段で解決）:
例: Notion vs Confluence（チームナレッジ管理）

間接競合（同じジョブを異なる手段で解決）:
例: Notion vs Excel+メール（情報共有・記録）

非消費（そのジョブをまだ解決していない）:
例: 小規模チームがまだチャットのみで管理
→ 最大の市場機会は「非消費」に潜んでいる
```

### 機能パリティ vs 差別化の判断
| 機能タイプ | 戦略 | 理由 |
|---|---|---|
| 必須（なければ失注） | パリティ確保が先決 | 差別化の前提条件 |
| 差別化（勝因になる） | 深く投資 | 競合優位の源泉 |
| 付加価値（あると嬉しい） | 後回し | ROIが低い |
| 競合模倣（真似するだけ） | 慎重に判断 | 消耗戦・ブランド毀損リスク |

### プロダクト品質チェック（リリース前）
- [ ] ユーザーストーリーの受け入れ基準を全て満たしているか
- [ ] エラー状態・空状態・ローディング状態が全てデザインされているか
- [ ] モバイル/PC双方で動作確認済みか
- [ ] パフォーマンス目標（P99 < 1秒）を達成しているか
- [ ] アクセシビリティ基準（WCAG 2.1 AA）を確認したか

## §9 ASI時代のプロダクトマネジメント

> AGI到達後、「機能開発・バグ修正・ドキュメント作成」はAIが自律的に行う。PMの本質的価値が「何を作るか」の選択と「なぜ作るか」の問い立てに集約される。

### PM職能のASI耐性マトリクス

| PM機能 | 代替可能性(0-1) | 残存理由 |
|---|---|---|
| 仕様書・PRD作成 | 0.80 | AIがインタビュー内容から自動生成 |
| バックログ整理・チケット作成 | 0.85 | AIが課題から自動生成 |
| データ分析・レポート | 0.75 | AIがダッシュボードを自動更新 |
| A/Bテスト設計・分析 | 0.70 | AIが実験を自律設計・実行 |
| **PMF判断** | **0.25** | **「市場が本当に求めているか」の感覚はAIが苦手** |
| **ビジョン・方向性設定** | **0.15** | **「なぜ存在するか」の問いは人間の責任** |
| **倫理・社会的影響の判断** | **0.10** | **AGI時代に最も重要な問いは人間が担う** |

### ASI時代のPMF再定義

```
従来PMF: 製品が市場の課題を解決し、顧客が継続的に使ってくれる状態

ASI時代PMF: 以下の3層全てを満たす状態

Layer 1（機能PMF・従来と同じ）:
  - 課題を解決 → Retention・NPS・WTPで計測

Layer 2（AI時代PMF）:
  - AIエージェントが自律的に使えるか（API・ドキュメント・セルフサービス）
  - AI検索・AIエージェントに「選ばれる」プロダクト設計になっているか

Layer 3（信頼PMF・新設）:
  - ユーザーがプロダクトに「責任ある主体」を感じるか
  - データの取り扱いを信頼できるか（特にAIが処理するデータ）
  - 「このプロダクトを選ぶこと = 自分の価値観の表明」になっているか
```

### ASI時代のロードマップ設計原則

```
追加すべき問い（従来フレームワークへの拡張）:
□ この機能をAIが自動化したら何が残るか？ → 残る価値が本質
□ 2年後、競合がAGIでこの機能を作ったとき、自社の差別化は何か？
□ この機能でユーザーの「どんな人間的な体験」を守るか？
□ AIが代替した後でも「ユーザーがこのプロダクトを選ぶ理由」は何か？

ロードマップ優先順位の新軸:
  従来: Impact × Confidence / Effort（RICE）
  ASI追加: × ASI耐性スコア（この機能はAIに奪われないか？）
            × Trust Contribution（この機能がブランド信頼を高めるか？）
```

### Bezos Working Backwards × ASI時代のPRD

```
Press Release（プレスリリース）テンプレートへの追加:
  従来: 「[製品名]は[対象顧客]の[問題]を[解決策]で解決します」

  ASI追加欄:
  - 「この製品をAIエージェントが使うとしたら、どのAPIを呼ぶか？」
  - 「この製品をAIが代替した後でも、人間が使い続ける理由は何か？」
  - 「ユーザーのデータをAIが処理することへの信頼をどう設計するか？」
```
