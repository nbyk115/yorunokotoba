# App Design Patterns: モバイルアプリ設計標準

## 概要
モバイルアプリ（iOS/Android/クロスプラットフォーム）に特化した設計パターン・UXガイドライン・技術選定基準。
Webとは異なるアプリ固有の設計判断を構造化し、ux-designer・frontend-dev・growth-hackerが参照する。

---

## 1. プラットフォーム設計原則

### iOS（Human Interface Guidelines準拠）
- **一貫性**: システム標準コンポーネント（UIKit/SwiftUI）を優先。独自UIは最小限
- **直接操作**: タップ・スワイプ・ピンチで直感的に操作できる設計
- **フィードバック**: 全操作に視覚的・触覚的フィードバック（Haptics）を返す
- **メタファー**: 現実世界のメタファーを活用（スライダー、トグル、カード）
- **Safe Area**: ノッチ・Dynamic Island・ホームインジケーターを考慮

### Android（Material Design 3準拠）
- **Material You**: ダイナミックカラー対応（ユーザーの壁紙からカラー生成）
- **Adaptive Layout**: 折りたたみ・タブレット・Chromebookの画面サイズ対応
- **Navigation**: NavHost + Navigation Component。バックスタック管理
- **Edge-to-Edge**: ステータスバー・ナビゲーションバーの背景透過
- **予測バック**: Android 14+の予測的バックジェスチャー対応

### 共通原則
- **オフラインファースト**: ネットワークなしでも基本機能が使える設計
- **パフォーマンス**: 起動1秒以内、画面遷移300ms以内、60fps維持
- **バッテリー**: バックグラウンド処理の最小化、位置情報の適切な精度選択
- **アクセシビリティ**: VoiceOver/TalkBack対応、最小タップターゲット44pt/48dp

## 2. ナビゲーションパターン

### パターン選定ガイド

| パターン | 適用条件 | 例 |
|---|---|---|
| タブバー（下部） | 主要機能3-5個 | Instagram, LINE, よるのことば |
| ドロワー（ハンバーガー） | 機能が多い（6+）| Gmail, Slack |
| スタックナビゲーション | 階層的な画面遷移 | 設定画面, 詳細→サブ詳細 |
| モーダル/ボトムシート | 一時的なアクション | フィルター, 共有, 新規作成 |
| フルスクリーンモーダル | 没入体験 | カメラ, エディタ |

### タブバー設計ルール
- アイテム数: **3-5個**（6個以上は「その他」で隠す）
- アイコン: ラベル付き（アイコンだけでは伝わらない）
- アクティブ状態: 色変化 + アイコン変形（filled/outlined切り替え）
- バッジ: 未読数は最小限に（バッジ疲れを防ぐ）

### ジェスチャー設計
| ジェスチャー | 用途 | 注意 |
|---|---|---|
| スワイプバック | 前の画面に戻る | iOS標準。Android 14+も対応 |
| プルダウン | リフレッシュ | ローディングインジケーター必須 |
| 長押し | コンテキストメニュー | Haptics フィードバック必須 |
| スワイプ（リスト） | 削除・アーカイブ | 取り消し(Undo)を必ず用意 |
| ピンチ | ズーム | 画像・マップのみ |
| ダブルタップ | いいね・ズーム | 機能が明白な場合のみ |

## 3. アプリ特有UXパターン

### オンボーディング
```
パターン選定:
├─ 機能が自明（SNS等） → オンボーディングなし。即利用開始
├─ 初回設定が必要（プロフィール等） → プログレッシブオンボーディング
│   └─ 3ステップ以内。スキップ可能。後から設定変更可能
└─ 独自UIがある → コーチマーク（使用時に1箇所ずつ説明）
    └─ 全画面チュートリアルは禁止（離脱率が高い）
```

### プッシュ通知設計
- **許可リクエストのタイミング**: 初回起動時に出さない。価値を体験した後に出す
- **例**: 占い結果を見た後に「毎朝の運勢をお届けしますか？」
- **頻度**: 1日1-2回が上限。それ以上はunsubscribeを誘発
- **パーソナライズ**: 全員同じ通知は禁止。行動・設定に基づいて出し分け
- **ディープリンク**: 通知タップで該当画面に直接遷移

### オフラインファースト設計
```
データ戦略:
├─ Core Data（頻繁に参照）→ ローカルDB（SQLite/Realm/Hive）に保存
├─ Transient Data（一時的）→ メモリキャッシュ
├─ Remote Data（サーバー依存）→ キャッシュ + 同期キュー
└─ 同期コンフリクト → Last-Write-Wins or マージ戦略を事前定義

オフライン時のUI:
├─ オフラインバナー（非侵襲的に表示）
├─ 操作はキューに入れ、復帰時に自動送信
└─ 送信失敗 → リトライ3回 → ユーザーに通知
```

### Haptics（触覚フィードバック）
| 操作 | Haptics種別 | iOS | Android |
|---|---|---|---|
| ボタンタップ | Light Impact | UIImpactFeedbackGenerator(.light) | HapticFeedbackConstants.KEYBOARD_TAP |
| 切り替え（トグル） | Medium Impact | .medium | CONTEXT_CLICK |
| 削除・エラー | Notification Error | UINotificationFeedbackGenerator(.error) | REJECT |
| 成功・完了 | Notification Success | .success | CONFIRM |
| スクロール吸着 | Selection | UISelectionFeedbackGenerator | CLOCK_TICK |

## 4. 技術選定ガイド

### フレームワーク選定マトリクス

| 基準 | React Native | Flutter | SwiftUI (iOS) | Jetpack Compose (Android) |
|---|---|---|---|---|
| クロスプラットフォーム | ✅ iOS+Android | ✅ iOS+Android+Web | ❌ iOSのみ | ❌ Androidのみ |
| パフォーマンス | 良好（Bridge→JSI） | 優秀（Skia直描画） | 最良（ネイティブ） | 最良（ネイティブ） |
| 開発速度 | 速い（JS/TS既存知識） | 速い（Hot Reload） | 中（Apple専用） | 中（Google専用） |
| エコシステム | 最大（npm） | 成長中（pub.dev） | Apple純正 | Google純正 |
| 採用市場 | 最大 | 成長中 | iOS限定 | Android限定 |
| 推奨条件 | Webチームがある | UI重視・高性能 | iOSのみ | Androidのみ |

### 判断フロー
```
iOSとAndroid両方必要？
├─ Yes → チームにJS/TS経験者がいる？
│   ├─ Yes → React Native
│   └─ No → Flutter
└─ No → どちらか片方？
    ├─ iOS → SwiftUI
    └─ Android → Jetpack Compose
```

### ローカルデータベース選定
| DB | 用途 | プラットフォーム |
|---|---|---|
| SQLite + Drift | 構造化データ・リレーション | Flutter |
| Realm | オブジェクトDB・リアルタイム同期 | React Native / Swift / Kotlin |
| MMKV | Key-Value（設定・キャッシュ） | React Native |
| Core Data | Apple純正ORM | SwiftUI |
| Room | Android純正ORM | Jetpack Compose |
| Hive | 軽量Key-Value | Flutter |

## 7. PWA vs ネイティブ判断基準

| 基準 | PWA推奨 | ネイティブ推奨 |
|---|---|---|
| ハードウェアアクセス | カメラ・位置情報程度 | Bluetooth・NFC・ARKit・HealthKit |
| オフライン要件 | 基本的なキャッシュ | 複雑なローカルDB・同期 |
| プッシュ通知 | Web Push で十分 | リッチ通知・サイレント通知が必要 |
| パフォーマンス | 60fps不要 | ゲーム・アニメーション・動画処理 |
| 配布 | URL共有でOK | ストア掲載が必要（ブランド信頼性） |
| 課金 | 自前決済OK | アプリ内課金（IAP）が必要 |
| 予算 | 低〜中 | 中〜高 |
| **よるのことば** | **現状PWAで正解** | ネイティブ化は将来の選択肢 |

## 8. Web → アプリ化の実装パス選定

> 既に Web / PWA で稼働中のサービスをモバイルアプリ化する判断と実装ルートを定義する。L188-199「PWA vs ネイティブ判断基準」で「ネイティブ推奨」と判定された場合の次工程。

### 8.1 判断フロー（上から順に評価し、満たせなければ次の段へ）

1. PWA で十分か（L188-199 のテーブル再評価）
   - ハードウェア要件 / IAP / ストア審査 / リッチ通知 のいずれも不要なら「PWA 据え置き + ホーム画面追加訴求」で完結。アプリ化しない判断が最強。
2. Capacitor / PWABuilder で WebView ラップ可能か
   - 既存 PWA をストア配布したいだけなら Capacitor が最短。Web コードベースを 100% 流用、追加学習コスト最小。
3. Expo（React Native）で書き換える価値があるか
   - ネイティブ UI / 高頻度更新 / OTA 配信が必要なら Expo。EAS Build でストア提出も自動化。
4. React Native / Flutter で本格ネイティブ化するか
   - パフォーマンス要件が厳しい、独自ネイティブモジュール多数、長期運用コミット可能なら採用。
5. Shipper.now 等の AI 生成 SaaS で試作するか
   - PoC / 社内デモ / クライアント提案前のたたき台。本番配布の主軸には現時点で推奨しない（8.3 参照）。

判断は「下に降りるほど工数 / lock-in / 学習コスト増」。Strangler Fig 的に PWA → Capacitor → Expo の順で段階移行できる設計を最優先。

### 8.2 実装パス比較表

| 評価軸 | Shipper.now | Expo（RN） | React Native（素） | Capacitor | Tauri Mobile |
|---|---|---|---|---|---|
| 初期コスト | 極低（URL 1 本） | 低（テンプレ豊富） | 中（環境構築要） | 低（Web 流用） | 中（Rust 学習） |
| 開発速度 | 最速（生成型） | 速い | 標準 | 速い（Web 同等） | 遅い（事例少） |
| カスタマイズ性 | 低（ブラックボックス） | 中〜高 | 最高 | 中（WebView 制約） | 高（要 Rust） |
| lock-in リスク | 高（SaaS 依存） | 中（Expo 依存） | 低（OSS 標準） | 低（OSS 標準） | 低（OSS） |
| ストア審査通過率 | 不明（実績不足） | 高（実績多） | 高（実績最多） | 中（WebView 拒否例あり） | 不明（事例少） |
| 多言語対応 | 不明 | i18next 等で標準 | i18next 等で標準 | Web の i18n 流用 | OSS i18n 利用可 |

評価ラベル: 「初期コスト極低」「ストア審査通過率高」等は INFERENCE（公式ドキュメント / コミュニティ事例から推定）。Shipper.now の「ストア審査通過率」「多言語対応」は SPECULATION（公開 1 週間、独立検証データなし）。

### 8.3 Shipper.now の評価（FACT / INFERENCE / SPECULATION）

- FACT: 2026-04-29 公開、URL 1 本入力でネイティブアプリを生成すると公式が告知（出典: 一次情報の独立確認は 2026-05-04 時点で未完了、要 WebFetch 検証）。
- FACT: 生成エンジンに Claude Opus 4.7 を採用と公式が表明（同上、要一次情報確認）。
- INFERENCE: URL → アプリ変換型 SaaS は WebView ラッパー寄りの構造になりがちで、UI ネイティブ感は Capacitor 同等の制約を受ける可能性が高い。
- SPECULATION: 「Shipper が Expo より優れている」「ストア審査を確実に通せる」等の優位性主張は 2026-05-04 時点で実績不足のため断定不可。本番採用前に最低 1 アプリで PoC → ストア提出 → 30 日運用の検証必須。

採用ガイダンス: クライアント提案で第一候補に置かない。社内 PoC / 提案前のモック生成用途に限定し、本番は Capacitor / Expo を主軸として比較検討する。

### 8.4 「よるのことば」適用シミュレーション

前提: 現状 PWA 稼働中、占いコンテンツ配信、ICP は深夜帯のスマホ占い利用層（参照: `ICP.md` L66）。

L188-199 テーブルで再評価:

| 基準 | よるのことば現状 | 判定 |
|---|---|---|
| ハードウェアアクセス | カメラ / 位置情報すら不要 | PWA 推奨 |
| オフライン要件 | 占いコンテンツのキャッシュ程度 | PWA 推奨 |
| プッシュ通知 | 深夜配信のリッチ通知が CVR を左右する可能性 | ネイティブ推奨 |
| 配布 | ストア掲載でブランド信頼性向上の余地 | ネイティブ推奨 |
| 課金 | 月額 / 都度課金を IAP 化すれば iOS 流入拡大 | ネイティブ推奨 |

判定: 5 軸中 3 軸が「ネイティブ推奨」に傾く。ただし即フル React Native 化はオーバーエンジニアリング。推奨パス:

- フェーズ 1（0-1 ヶ月）: Capacitor で既存 PWA をラップし iOS / Android 両ストア提出。Web コードベース 100% 流用、追加開発工数最小。
- フェーズ 2（2-4 ヶ月）: ストア掲載後の DL 数 / リテンション / IAP 売上を計測。PWA 経由と比較。
- フェーズ 3（5 ヶ月以降）: Capacitor で性能 / UX 不足が顕在化したら Expo 移行を検討。Branch by Abstraction で画面単位に置換。

Shipper.now は本案件では非採用（占いという情緒価値主軸サービスで UI ネイティブ感の検証コストが高く、SaaS lock-in リスクと釣り合わない）。

## 9. AIネイティブテスト（iOSシミュレーター連携）

> **XCUITestのスクリプトを書く代わりに、Claude Codeがシミュレーター上のアプリを自律的にテストする。**

### 仕組み
- アクセシビリティツリーでUIの構造をセマンティックに把握（座標ベースではない）
- スクリーンショットとの併用で精度向上（構造データ+画像認識のハイブリッド）
- 1画面あたり10-50トークンで処理可能（コスト実用的）

### 導入方法
```bash
# ios-simulator-skillをClaude Codeスキルとして追加
# CLAUDE.mdにテストフローを事前定義
```

### テスト可能な範囲
| テスト種別 | 内容 |
|---|---|
| 全画面クロール | 全画面を自動遷移し、クラッシュ・表示崩れを検知 |
| フロー検証 | ログイン→メイン→設定→ログアウト等の操作フローを自動実行 |
| エッジケース | 未入力・長文・特殊文字・オフライン時の挙動 |
| アクセシビリティ | VoiceOver対応状況・タップターゲットサイズの検証 |

### 注意事項
- Xcodeシミュレーター環境が必要（PWAでは使えない）
- 大規模アプリでの実用性は要検証（デモはシンプルなアプリ）
- CLAUDE.mdにテストフローの事前定義が必要（メンテナンスコストはゼロではない）
- QAの自動化は「できるかどうか」→「どう運用に載せるか」のフェーズ

### mobai.run（Claudeプラグイン型バグテスト）

> **アプリ内の全潜在バグを自律テストし、発見を報告するClaudeプラグイン。**

- **導入**: mobai.run からプラグインインストール
- **特徴**: ios-simulator-skillと違い、プラグイン形式で即導入。手動セットアップ不要
- **用途**: アプリ全体の潜在バグ検出・レポート生成
- **使い分け**:
  - ios-simulator-skill: カスタムテストフロー・詳細制御が必要な場合
  - mobai.run: 「全部テストして」のワンショット型。初期スクリーニングに最適

## 禁止事項
- 全画面チュートリアル（離脱率が高い。コーチマークを使う）
- 初回起動時のプッシュ通知許可リクエスト（価値体験後に出す）
- 6個以上のタブバーアイテム
- Haptics なしの破壊的操作（削除・送信）
- オフライン時の無反応（エラー表示もフィードバックもなし）
- アプリ内課金の価格をハードコード（サーバーから取得）
- ストアガイドライン違反（審査リジェクトで公開不可）
