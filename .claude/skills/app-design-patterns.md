# App Design Patterns — モバイルアプリ設計標準

## 概要
モバイルアプリ（iOS/Android/クロスプラットフォーム）に特化した設計パターン・UXガイドライン・技術選定基準。
Webとは異なるアプリ固有の設計判断を構造化し、ux-designer・frontend-dev・growth-hackerが参照する。

---

## 1. プラットフォーム設計原則

### iOS（Human Interface Guidelines準拠）
- **一貫性**: システム標準コンポーネント（UIKit/SwiftUI）を優先。独自UIは最小限
- **直接操作**: タップ・スワイプ・ピンチで直感的に操作できる設計
- **フィードバック**: 全操作に視覚的・触覚的フィードバック（Haptics）を返す
- **メタファー**: 現実世界のメタファーを活用（スライダー、トグル、カード）
- **Safe Area**: ノッチ・Dynamic Island・ホームインジケーターを考慮

### Android（Material Design 3準拠）
- **Material You**: ダイナミックカラー対応（ユーザーの壁紙からカラー生成）
- **Adaptive Layout**: 折りたたみ・タブレット・Chromebookの画面サイズ対応
- **Navigation**: NavHost + Navigation Component。バックスタック管理
- **Edge-to-Edge**: ステータスバー・ナビゲーションバーの背景透過
- **予測バック**: Android 14+の予測的バックジェスチャー対応

### 共通原則
- **オフラインファースト**: ネットワークなしでも基本機能が使える設計
- **パフォーマンス**: 起動1秒以内、画面遷移300ms以内、60fps維持
- **バッテリー**: バックグラウンド処理の最小化、位置情報の適切な精度選択
- **アクセシビリティ**: VoiceOver/TalkBack対応、最小タップターゲット44pt/48dp

## 2. ナビゲーションパターン

### パターン選定ガイド

| パターン | 適用条件 | 例 |
|---|---|---|
| タブバー（下部） | 主要機能3-5個 | Instagram, LINE, よるのことば |
| ドロワー（ハンバーガー） | 機能が多い（6+）| Gmail, Slack |
| スタックナビゲーション | 階層的な画面遷移 | 設定画面, 詳細→サブ詳細 |
| モーダル/ボトムシート | 一時的なアクション | フィルター, 共有, 新規作成 |
| フルスクリーンモーダル | 没入体験 | カメラ, エディタ |

### タブバー設計ルール
- アイテム数: **3-5個**（6個以上は「その他」で隠す）
- アイコン: ラベル付き（アイコンだけでは伝わらない）
- アクティブ状態: 色変化 + アイコン変形（filled/outlined切り替え）
- バッジ: 未読数は最小限に（バッジ疲れを防ぐ）

### ジェスチャー設計
| ジェスチャー | 用途 | 注意 |
|---|---|---|
| スワイプバック | 前の画面に戻る | iOS標準。Android 14+も対応 |
| プルダウン | リフレッシュ | ローディングインジケーター必須 |
| 長押し | コンテキストメニュー | Haptics フィードバック必須 |
| スワイプ（リスト） | 削除・アーカイブ | 取り消し(Undo)を必ず用意 |
| ピンチ | ズーム | 画像・マップのみ |
| ダブルタップ | いいね・ズーム | 機能が明白な場合のみ |

## 3. アプリ特有UXパターン

### オンボーディング
```
パターン選定:
├─ 機能が自明（SNS等） → オンボーディングなし。即利用開始
├─ 初回設定が必要（プロフィール等） → プログレッシブオンボーディング
│   └─ 3ステップ以内。スキップ可能。後から設定変更可能
└─ 独自UIがある → コーチマーク（使用時に1箇所ずつ説明）
    └─ 全画面チュートリアルは禁止（離脱率が高い）
```

### プッシュ通知設計
- **許可リクエストのタイミング**: 初回起動時に出さない。価値を体験した後に出す
- **例**: 占い結果を見た後に「毎朝の運勢をお届けしますか？」
- **頻度**: 1日1-2回が上限。それ以上はunsubscribeを誘発
- **パーソナライズ**: 全員同じ通知は禁止。行動・設定に基づいて出し分け
- **ディープリンク**: 通知タップで該当画面に直接遷移

### オフラインファースト設計
```
データ戦略:
├─ Core Data（頻繁に参照）→ ローカルDB（SQLite/Realm/Hive）に保存
├─ Transient Data（一時的）→ メモリキャッシュ
├─ Remote Data（サーバー依存）→ キャッシュ + 同期キュー
└─ 同期コンフリクト → Last-Write-Wins or マージ戦略を事前定義

オフライン時のUI:
├─ オフラインバナー（非侵襲的に表示）
├─ 操作はキューに入れ、復帰時に自動送信
└─ 送信失敗 → リトライ3回 → ユーザーに通知
```

### Haptics（触覚フィードバック）
| 操作 | Haptics種別 | iOS | Android |
|---|---|---|---|
| ボタンタップ | Light Impact | UIImpactFeedbackGenerator(.light) | HapticFeedbackConstants.KEYBOARD_TAP |
| 切り替え（トグル） | Medium Impact | .medium | CONTEXT_CLICK |
| 削除・エラー | Notification Error | UINotificationFeedbackGenerator(.error) | REJECT |
| 成功・完了 | Notification Success | .success | CONFIRM |
| スクロール吸着 | Selection | UISelectionFeedbackGenerator | CLOCK_TICK |

## 4. 技術選定ガイド

### フレームワーク選定マトリクス

| 基準 | React Native | Flutter | SwiftUI (iOS) | Jetpack Compose (Android) |
|---|---|---|---|---|
| クロスプラットフォーム | ✅ iOS+Android | ✅ iOS+Android+Web | ❌ iOSのみ | ❌ Androidのみ |
| パフォーマンス | 良好（Bridge→JSI） | 優秀（Skia直描画） | 最良（ネイティブ） | 最良（ネイティブ） |
| 開発速度 | 速い（JS/TS既存知識） | 速い（Hot Reload） | 中（Apple専用） | 中（Google専用） |
| エコシステム | 最大（npm） | 成長中（pub.dev） | Apple純正 | Google純正 |
| 採用市場 | 最大 | 成長中 | iOS限定 | Android限定 |
| 推奨条件 | Webチームがある | UI重視・高性能 | iOSのみ | Androidのみ |

### 判断フロー
```
iOSとAndroid両方必要？
├─ Yes → チームにJS/TS経験者がいる？
│   ├─ Yes → React Native
│   └─ No → Flutter
└─ No → どちらか片方？
    ├─ iOS → SwiftUI
    └─ Android → Jetpack Compose
```

### ローカルデータベース選定
| DB | 用途 | プラットフォーム |
|---|---|---|
| SQLite + Drift | 構造化データ・リレーション | Flutter |
| Realm | オブジェクトDB・リアルタイム同期 | React Native / Swift / Kotlin |
| MMKV | Key-Value（設定・キャッシュ） | React Native |
| Core Data | Apple純正ORM | SwiftUI |
| Room | Android純正ORM | Jetpack Compose |
| Hive | 軽量Key-Value | Flutter |

## 5. アプリKPI・グロース指標

### 北極星指標の候補
| アプリ種別 | 北極星指標 | 理由 |
|---|---|---|
| SNS | DAU/MAU比率 | エンゲージメントの健全性 |
| EC | 購入完了数/月 | 直接的な収益指標 |
| SaaS | Weekly Active Users | 定着度 |
| コンテンツ | セッション時間/日 | エンゲージメント深度 |
| 占い・エンタメ | D7リテンション | 習慣化の指標 |

### リテンション基準（業界平均）
| 期間 | 優秀 | 平均 | 要改善 |
|---|---|---|---|
| D1（翌日） | 40%+ | 25-35% | 25%未満 |
| D7（1週間） | 20%+ | 12-18% | 12%未満 |
| D30（1ヶ月） | 10%+ | 5-8% | 5%未満 |

### ASO（App Store Optimization）
| 要素 | iOS (App Store) | Android (Google Play) |
|---|---|---|
| タイトル | 30文字。キーワード含める | 30文字。キーワード含める |
| サブタイトル | 30文字 | なし（短い説明80文字） |
| キーワードフィールド | 100文字（カンマ区切り） | なし（説明文から自動抽出） |
| スクリーンショット | 最大10枚。縦型推奨 | 最大8枚。縦型推奨 |
| プレビュー動画 | 最大3本（30秒） | 1本 |
| レーティング | 4.0以上を維持 | 4.0以上を維持 |
| レビュー返信 | 全ネガティブに48時間以内返信 | 同左 |

## 6. アプリ固有のセキュリティ

### チェックリスト
- [ ] APIキーをアプリバイナリに埋め込まない（サーバー経由で取得）
- [ ] 証明書ピニング（SSL Pinning）の実装
- [ ] ルート/Jailbreak検知
- [ ] スクリーンショット防止（金融・機密情報画面）
- [ ] バイオメトリック認証（Face ID / Touch ID / 指紋）の適切な実装
- [ ] KeychainStore / Android Keystore でトークン保存
- [ ] ProGuard/R8（Android）でコード難読化
- [ ] App Transport Security（iOS）の適切な設定

## 7. PWA vs ネイティブ判断基準

| 基準 | PWA推奨 | ネイティブ推奨 |
|---|---|---|
| ハードウェアアクセス | カメラ・位置情報程度 | Bluetooth・NFC・ARKit・HealthKit |
| オフライン要件 | 基本的なキャッシュ | 複雑なローカルDB・同期 |
| プッシュ通知 | Web Push で十分 | リッチ通知・サイレント通知が必要 |
| パフォーマンス | 60fps不要 | ゲーム・アニメーション・動画処理 |
| 配布 | URL共有でOK | ストア掲載が必要（ブランド信頼性） |
| 課金 | 自前決済OK | アプリ内課金（IAP）が必要 |
| 予算 | 低〜中 | 中〜高 |
| **よるのことば** | **現状PWAで正解** | ネイティブ化は将来の選択肢 |

## 禁止事項
- 全画面チュートリアル（離脱率が高い。コーチマークを使う）
- 初回起動時のプッシュ通知許可リクエスト（価値体験後に出す）
- 6個以上のタブバーアイテム
- Haptics なしの破壊的操作（削除・送信）
- オフライン時の無反応（エラー表示もフィードバックもなし）
- アプリ内課金の価格をハードコード（サーバーから取得）
- ストアガイドライン違反（審査リジェクトで公開不可）
