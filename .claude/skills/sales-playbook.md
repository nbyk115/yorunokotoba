# Sales Playbook — 営業戦略・商談設計・クロージング

> **参照エージェント**: lead-qualifier, proposal-writer, client-success, strategy-lead
> **用途**: インサイドセールス・フィールドセールス・エンタープライズ商談・SaaSサブスクリプション営業

---

## §1 売れる構造の設計原則

### 属人営業からの脱却
```
❌ 属人営業（スケールしない）:
  - トップ営業1人が全案件をクロージング
  - 「信頼関係」だけで受注している
  - 他のメンバーが再現できない

✅ 構造型営業（スケールする）:
  - ICP（Ideal Customer Profile）で対象を絞る
  - 標準化されたヒアリング→提案→クロージングフロー
  - KPIで営業ファネルを管理（ARR・ACV・ASP・CVR）
```

### 佐藤裕介の売り方原則
1. **売りつけない**: 「なぜ今・なぜ我々か」に顧客自身が答えられる状態を作る
2. **構造で売る**: 市場構造・競合比較・ROI試算が自動的に購買理由になるよう設計
3. **再現性最優先**: 1人の天才より、誰でも70点を出せるプロセスが会社を強くする

## §2 ICP定義（Ideal Customer Profile）

### ICPフレームワーク
```
【ファームグラフィクス（企業属性）】
- 業種: [最も受注確率が高い業種Top3]
- 従業員数: [最も受注確率が高い規模]
- 売上規模: [ターゲットARR帯]
- 地域: [日本国内 / 海外 / 特定都市]

【テクノグラフィクス（技術スタック）】
- 利用中のツール（競合 / 補完製品）
- デジタル成熟度

【シグナル（購買タイミング）】
- 採用情報（特定ポジションの募集中）
- 資金調達・組織拡大ニュース
- 競合製品の解約シグナル
- 痛みの発生イベント（コンプライアンス変更・組織再編）

【決裁構造】
- 最終決裁者（CxO / 事業部長）
- 推進者（現場責任者）
- 評価者（IT部門 / 法務）
- 妨害者（変化を嫌う既存ベンダー）
```

### ICPスコアリング（Lead Score）
| 項目 | 配点 | 判定基準 |
|---|---|---|
| 業種適合 | 30点 | Top業種: 30点 / 関連業種: 15点 / 対象外: 0点 |
| 規模適合 | 20点 | Best规模: 20点 / 許容範囲: 10点 / 対象外: 0点 |
| タイミングシグナル | 30点 | 明確なシグナルあり: 30点 / 弱いシグナル: 15点 |
| 決裁者アクセス | 20点 | 決裁者に直接アクセス可: 20点 / 現場のみ: 5点 |
| **合計** | **100点** | **70点以上 = 優先商談** |

## §3 ヒアリング設計（MEDDPICC）

```
M — Metrics（成功指標）
  「この課題が解決したとき、どんな数字で成功を測りますか？」
  → 顧客のKPIを数値で合意する

E — Economic Buyer（経済的決裁者）
  「最終的な予算承認をする方はどなたですか？」
  → 決裁者への直接アクセスを確保する

D — Decision Criteria（意思決定基準）
  「ベンダーを選ぶ際の評価基準は何ですか？優先順位は？」
  → 提案の勝ち筋を把握する

D — Decision Process（意思決定プロセス）
  「今後の意思決定のスケジュールはどうなっていますか？」
  → 稟議・評価・承認の全ステップを把握

P — Paper Process（契約プロセス）
  「契約の際に必要な手続きや審査はありますか？」
  → リーガル・セキュリティ審査の工数を事前把握

I — Identify Pain（課題の特定）
  「現状の最大の痛みは何ですか？それを放置するとどうなりますか？」
  → 課題の緊急性・深刻度を数値化

C — Champion（社内推進者）
  「社内で我々のソリューションを推進してくれる方は誰ですか？」
  → 社内チャンピオンを作り、内部稟議を支援する

C — Competition（競合状況）
  「現在、他のソリューションと比較していますか？」
  → 勝ち筋と差別化ポイントを確認
```

## §4 提案書の構造（SPIN×PL）

### 勝てる提案書の4層構造
```
Layer 1: 顧客の現状と痛みを言語化（Situation + Problem）
  - 顧客の業界・競合環境・現状KPI
  - 「今、XX という問題が起きている」を顧客より詳しく説明する
  - → 「我々をよく研究している」という信頼を勝ち取る

Layer 2: 放置コストの数値化（Implication）
  - 「この問題を解決しないと年間XXX万円のロスになる」
  - コストは「機会損失」「人件費」「競合に奪われるシェア」で計算する

Layer 3: ソリューションとROI（Need-payoff）
  - 導入後の改善インパクトを数値で示す（3シナリオ）
  - 回収期間（ペイバック月数）を明示する

Layer 4: 次のアクション（Go/No-Go）
  - 提案の終わりに「次回のステップ」を必ず合意する
  - 「検討します」で終わらせない
```

### ROI試算テンプレート
```
【現状コスト】
  - 課題Aによるコスト: ____万円/月
  - 課題Bによるコスト: ____万円/月
  - 合計: ____万円/月（____万円/年）

【導入費用】
  - 初期費用: ____万円
  - 月額費用: ____万円/月
  - 年間総費用: ____万円/年

【改善効果（標準シナリオ）】
  - 効果A: ____万円/月削減
  - 効果B: ____万円/月増収
  - 合計改善効果: ____万円/月

【ROI計算】
  ペイバック期間: ____ヶ月
  1年ROI: ____%
  3年累積効果: ____万円
```

## §5 クロージング戦術

### クロージングの前提条件チェック
```
□ Champion（社内推進者）が確保されているか
□ 決裁者に提案内容が届いているか
□ ROI試算に顧客が合意しているか
□ 競合比較で優位性が認識されているか
□ 稟議・リーガルレビューの完了見込みが把握できているか
□ 価格交渉の余地と最低ラインが決まっているか
```

### クロージング失敗パターンと対処
| パターン | 原因 | 対処 |
|---|---|---|
| 「検討します」 | 決裁者に届いていない | Champion に「役員プレゼン」を依頼 |
| 「価格が高い」 | ROIが伝わっていない | ペイバック計算を共同で実施 |
| 「今は時期が悪い」 | 痛みが十分でない | 放置コストを再計算して提示 |
| 「競合と迷っている」 | 差別化が弱い | 比較表を作り、勝てる評価軸を定義 |
| 「評価が長引く」 | Paper processを把握していなかった | 稟議フローを初回で確認する |

## §6 SaaSサブスクリプション特有の指標

### 営業KPIツリー
```
ARR（年間経常収益）
├─ 新規ARR
│   ├─ 商談数 × 受注CVR × ACV
│   └─ リードソース別ROI（MQL→SQL→受注）
├─ 拡張ARR（既存顧客のアップセル/クロスセル）
│   └─ NRR（Net Revenue Retention）
└─ チャーンARR（解約分）
    └─ Gross Revenue Retention
```

### セールスサイクル最適化
| ステージ | 目標日数 | アクション |
|---|---|---|
| リード → MQL | 3日以内 | スコアリング自動化 |
| MQL → SQL | 5営業日以内 | SDRによる資格確認 |
| SQL → 提案 | 10営業日以内 | AEによるディスカバリー + 提案 |
| 提案 → クロージング | 20〜45日 | POC/トライアル → 稟議支援 |

## §7 品質チェック（提案・商談前）

```
□ ICPスコア 70点以上の案件か
□ MEDDPICC の全項目を把握しているか
□ ROI試算に数値根拠があるか（「大幅改善」は禁止）
□ 決裁者に直接アクセスできているか
□ 次のアクション（日時・担当）が合意されているか
□ 競合状況を把握し、差別化ポイントが明確か
□ paper processの工数・スケジュールが把握できているか
```

## 🔺 反証モード（トリプルチェック必須）

本スキルを適用する全アウトプットは、CLAUDE.md「反証モード」セクションに定義されたトリプルチェックを必ず実行すること。

1. **自己反証**: 結論に対する反論・反例を最低3つ挙げ、確証バイアスを排除
2. **構造反証**: ロジックの飛躍・数値の妥当性・抜け漏れを検証
3. **実用反証**: 実行可能性・エッジケース・最悪シナリオを検証

チェックなしのアウトプットはドラフト扱い。省略禁止。
