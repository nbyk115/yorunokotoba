# output-quality-rubrics: 出力タイプ別品質ルーブリック

## 用途

出力タイプ別 100 点満点採点フレーム。brand-guardian の「PASS / REJECT 二択」を補完する中間グラデーション層、80 点未満で具体的修正指示自動生成。

## ルーブリック 5 種

### 1. 提案書 / Pitch Deck (Markdown 設計層 or HTML 配信層)

| 軸 | 配点 | 評価基準 |
|---|---|---|
| 顧客課題の構造化 | 25 | 顧客の本質課題が ICP.md 準拠で明示、曖昧な要望 → 業務要件 / 技術要件への落とし込み |
| PL インパクト数値化 | 20 | 粗利インパクト / ブレイクイーブン / LTV/CAC / 感度分析の 4 項目すべて記載 |
| 競合差別化 | 15 | ExaWizards / Sierra / 情報戦略テクノロジー等の競合分析 + 自社 vertical 優位明示 |
| 出典明記 | 15 | FACT / INFERENCE / SPECULATION 3 ラベル全件付与、出典なし数値断言ゼロ |
| 反証チェック付与 | 10 | Step 1-4 全件付与 (自己反証 / 構造反証 / 実用反証 / リスク即潰し) |
| 日本語字形 / ブランド規律 | 10 | ハードルール 10 (lang ja / Noto Sans JP) + ハードルール 16 (em-dash 0 / raw ** 0) |
| 視覚的構造化 | 5 | HTML 配信層 = brand-guardian 5 項目 PASS / Markdown 設計層 = 表 + 階層明確 |

80 点未満時の具体的修正指示:
- 課題構造化不足 → ICP.md 参照 + 「曖昧 → 要件」変換テンプレ適用
- 数値根拠不足 → revenue-growth-framework §1-4 PL チェックリスト適用
- 競合分析不足 → competitive-analyst agent 起動

### 2. visual / IR デッキ (HTML 1280×720)

| 軸 | 配点 | 評価基準 |
|---|---|---|
| 情報密度 | 25 | 1 枚絵で全体構造把握可能、参照ライブラリ 3-5 件 query 済（DESIGN.md §12） |
| デザイン基準 | 20 | DESIGN.md カラー / フォント / スペーシング準拠 + 業界別参照パターン適用 |
| ブランド規律 5 項目 | 20 | lang ja / charset UTF-8 / Noto Sans JP / em-dash 0 / raw ** 0 |
| 視覚的階層 | 15 | 主要メッセージの優先順位明確、視線誘導設計 |
| インタラクティブ性 | 10 | クリック / hover / タブ等の操作要素、HTML-First 思考術 (PR #140) 適用 |
| 数値・出典の visualization | 10 | チャート / 数値の出典 + INFERENCE/FACT ラベル可視化 |

80 点未満時:
- 情報密度不足 → Lazyweb For Humans / Canva MCP / refero.design / WebSearch で参照取得 (DESIGN.md §12.3)
- ブランド規律違反 → brand-guardian 5 項目検証コマンド実行

### 3. コード (skill / agent / hook / フロントエンド等)

| 軸 | 配点 | 評価基準 |
|---|---|---|
| 外科的変更 (CLAUDE.md §4) | 25 | 依頼範囲のみ、隣接物への「ついで」変更ゼロ |
| Karpathy 12 ルール準拠 | 20 | 12 ルール (PR #138) 全件遵守、特にルール 6 トークン予算 + ルール 10 チェックポイント |
| シンプルさ (Boris #3) | 15 | 最小限のコード、抽象化過剰なし、シニアエンジニア視点で複雑すぎない |
| テスト / 検証 | 15 | typecheck / test / lint 即実行、テストが意図を検証（ルール 9） |
| コメント最小化 | 10 | コメント = WHY のみ、自明な WHAT は書かない |
| 規律準拠 (ハードルール 6/7/12/13/16/17) | 15 | PR 経由 / Squash merge / 1 コミット 1 目的 / em-en dash 0 / 主語詐称なし |

80 点未満時:
- 外科的変更違反 → diff 差し戻し、依頼範囲再確認
- テスト不在 → `/tdd` skill 起動

### 4. 戦略 / コンサル文書

| 軸 | 配点 | 評価基準 |
|---|---|---|
| 第一原理分解 | 25 | TAM / SAM / SOM + VRIO + 勝ちパターン分類 (revenue-growth-framework §4) |
| マスク式 4 原則準拠 | 15 | インセンティブ / 評判 / Unkind Truth / アルゴリズム全件考慮 |
| アルトマン式 10 原則準拠 | 15 | 該当原則の明示的参照 (特に原則 3 垂直特化 / 9 現実主義) |
| 出典 + 反証チェック | 15 | FACT/INFERENCE/SPECULATION + Step 1-4 |
| 関根さん / 水野さん適用パス | 15 | 既存案件への即適用パス明示 |
| 体系統合 (75 原則 + Defensibility 7 軸との整合) | 15 | 既存体系との接続点明示、独立性 / 重複の honest 評価 |

### 5. 反証チェック自体の品質

| 軸 | 配点 | 評価基準 |
|---|---|---|
| Step 1 自己反証 | 25 | INFERENCE / SPECULATION 明示、断定回避 |
| Step 2 構造反証 | 25 | 既存体系 + ハードルール + 規律との整合検証 |
| Step 3 実用反証 | 25 | 実測コマンド + 実出力添付、narrative 単独は無効 |
| Step 4 リスク即潰し | 25 | 発見リスクの構造的発生不可能化 / 次セッション持ち越し理由明示 |

80 点未満時: 該当 Step を再実行、実測値併記。

## 起動条件

- ユーザーが「採点」「ルーブリック」「品質評価」発話時
- brand-guardian が「中間判定が必要」と判断した時
- 関根さん / 水野さん案件の納品物検証時

## 既存体系との接続

- brand-guardian 5 項目検証 (PR #140): 機械的 PASS / REJECT 層
- 反証チェック Step 1-4: メタ品質保証層
- 本 skill: 中間グラデーション層 = 80 点未満時の具体的修正指示自動生成

## 反証チェック（Step 1-4 圧縮）

- Step 1: ルーブリック内容は assistant 構築 INFERENCE / 配点バランスは経験則 INFERENCE
- Step 2: 既存 brand-guardian + 反証チェック + 75 原則体系と整合
- Step 3: 5 種ルーブリック全件具体評価軸 + 80 点未満時の修正指示明示
- Step 4 リスク即潰し: 「ルーブリック形骸化」リスクは起動条件明示 + 既存規律との接続点明示で構造的回避
