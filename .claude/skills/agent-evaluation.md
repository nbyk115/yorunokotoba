# Agent Evaluation — 自己評価・フィードバックループ・自動改善

## 概要
全エージェントが自らのアウトプットを定量評価し、成果データを蓄積し、スキルファイルの自動改善を駆動する**自己進化エンジン**。
「人間が指摘する前にエージェント自身が気づき、修正する」を実現する。

---

## 原則
- **計測なき改善なし**: 全アウトプットにスコアを付ける。感覚ではなくデータで進化する
- **小さく・速く・頻繁に**: 大規模リファクタリングではなく、1セッション1改善を積み重ねる
- **失敗は資産**: 低スコアのアウトプットこそ最大の学習源。記録し、分析し、再発防止する
- **自動化が前提**: 人間の介在なしで評価→診断→修正→検証のサイクルが回る

---

## 1. セッション後自己評価プロトコル

### 全エージェント必須: アウトプット評価カード

タスク完了時に以下を `Archival Memory` に書き込む:

```
【評価カード】
日付: YYYY-MM-DD
エージェント: [agent-name]
タスク種別: [S1/S2/S3/S4 のシナリオ番号]
参照スキル: [使用したスキルファイル名]

■ 品質スコア（各1-5, 合計25点満点）
  Q1 正確性:     [1-5] — 事実・数値・ロジックに誤りがないか
  Q2 具体性:     [1-5] — 抽象論ではなく具体的なアクション・数値があるか
  Q3 実行可能性:  [1-5] — クライアント/チームが実際に実行できるか
  Q4 PL直結度:   [1-5] — 粗利・売上・コストへの影響が明示されているか
  Q5 ブランド適合: [1-5] — トーン・品質・禁止表現ルールを遵守しているか
  合計: [XX/25]

■ 反証チェック通過率
  Step1 自己反証:   [PASS/PARTIAL/FAIL]
  Step2 構造反証:   [PASS/PARTIAL/FAIL]
  Step3 実用反証:   [PASS/PARTIAL/FAIL]

■ 成果フィードバック（後日記入）
  採用結果: [採用/修正後採用/却下/未確認]
  クライアント反応: [ポジティブ/ニュートラル/ネガティブ/未確認]
  数値成果: [受注額/CVR変化/削減コスト等の実数値 or 未計測]

■ 自己診断
  うまくいった点: [1文]
  改善すべき点:   [1文]
  スキル修正提案: [具体的な修正案 or なし]
```

### スコア閾値とアクション

| 合計スコア | 判定 | アクション |
|---|---|---|
| 22-25 | A（優秀） | パターンを `agent_learnings` に記録。スキルの成功パターンとして蓄積 |
| 18-21 | B（良好） | 改善点を1つ特定し次回セッションで適用 |
| 14-17 | C（要改善） | スキル修正提案を作成 → `/evolve` で改善サイクルを起動 |
| 13以下 | D（問題あり） | 即座にスキル修正を実行。根本原因分析を `debug-methodology` で実施 |

## 2. フィードバックループエンジン

### 自動集計サイクル

```
Weekly Review（毎週自動実行）
├─ Archival Memory から直近7日の評価カードを収集
├─ エージェント別・スキル別の平均スコアを算出
├─ スコアが前週比で下降したスキルをフラグ
├─ 下降スキルに対して改善仮説を3つ生成
└─ evolution-log に記録

Monthly Review（毎月自動実行）
├─ 月次トレンド分析（エージェント別・スキル別・タスク種別）
├─ 成果フィードバック（採用率・クライアント反応）との相関分析
├─ スキルファイルの改訂候補をランキング
├─ A/Bテスト対象スキルを選定
└─ evolution-log に記録
```

### パフォーマンス低下の自動検知トリガー

以下の条件に該当した場合、**自動でスキル改善サイクルを起動**:

| トリガー | 条件 | アクション |
|---|---|---|
| スコア急落 | 平均スコアが前週比 -3以上 | 即座に原因分析 → スキル修正提案 |
| 連続低評価 | 同一スキルでC判定が3回連続 | スキルの構造的見直し → A/Bテスト候補に |
| 却下率上昇 | アウトプット却下率が30%以上 | `first-principles-breakdown` で根本原因分解 |
| 成果未達 | 数値成果が目標の50%未満 | strategy-lead に報告 → 戦略レベルで再検討 |

## 3. 成果トラッキングデータモデル

### エージェント別KPIダッシュボード

```
【エージェント成績表】
期間: YYYY-MM-DD 〜 YYYY-MM-DD

■ [agent-name]
  タスク実行数:       XX件
  平均品質スコア:     XX.X/25（前期比 +X.X）
  反証チェック通過率:  XX%
  採用率:            XX%（採用/修正後採用の合計）
  クライアント満足度:  ポジ XX% / ニュートラル XX% / ネガ XX%

  最も高スコアのタスク種別: [SX]（平均 XX.X）
  最も低スコアのタスク種別: [SX]（平均 XX.X）

  使用スキル別スコア:
    consulting-playbook:       XX.X（XX件）
    revenue-growth-framework:  XX.X（XX件）
    ...

  改善トレンド: [↑上昇中 / →横ばい / ↓下降中]
```

### スキル別パフォーマンスマップ

```
【スキル成績表】
期間: YYYY-MM-DD 〜 YYYY-MM-DD

■ [skill-name] v[X.X]
  参照回数:     XX件
  平均貢献スコア: XX.X/25
  使用エージェント: [agent1, agent2, ...]
  
  最も効果的な適用: [タスク種別 + 概要]
  最も効果が薄い適用: [タスク種別 + 概要]
  
  A/Bテスト履歴:
    v1.0 vs v1.1: v1.1が +2.3ポイント → v1.1採用（YYYY-MM-DD）
  
  次回改訂候補: [Yes/No]
  改訂仮説: [具体的な改善ポイント]
```

## 4. 自動スキル修正プロトコル

### 修正の3段階

```
Level 1: マイクロ修正（即時実行可）
├─ 禁止表現の追加・修正
├─ チェックリスト項目の追加
├─ 出力フォーマットの微調整
├─ 参照スキルの追加・変更
└─ 影響範囲: 当該スキルのみ

Level 2: セクション改訂（A/Bテスト推奨）
├─ シナリオプレイブックの書き換え
├─ フレームワークの追加・変更
├─ 干渉原則の適用方法の修正
└─ 影響範囲: 当該スキル + 参照エージェント

Level 3: 構造的リファクタリング（strategy-lead承認必須）
├─ スキルの分割・統合
├─ エージェント間の責務再定義
├─ 新スキルの追加
└─ 影響範囲: 複数部門
```

### 自動修正フロー

```
1. トリガー検知（スコア低下/却下率上昇）
2. 原因仮説の生成（最低3つ）
   └─ debug-methodology の反証ベースで仮説を検証
3. 修正案の作成
   ├─ Level 1 → 即時適用（Archival Memoryに変更理由を記録）
   ├─ Level 2 → A/Bテスト版を生成 → skill-evolution で検証
   └─ Level 3 → strategy-lead に提案書を提出
4. 修正の実行
5. 効果測定（修正前後の品質スコア比較）
6. evolution-log に全過程を記録
```

| 部門 | Q1正確性 | Q2具体性 | Q3実行可能性 | Q4 PL直結 | Q5ブランド |

| Product | ×1.0 | ×1.5 | ×1.0 | ×1.0 | ×0.5 |

```
【進化ログ】#[連番]
日付: YYYY-MM-DD
種別: [自動検知/手動トリガー/定期レビュー]
対象: [スキル名 or エージェント名]

■ 検知された問題
  指標: [スコア低下/却下率上昇/成果未達]
  データ: [具体的な数値]
  期間: [いつからいつまで]

■ 原因分析
  仮説1: [内容] → [検証結果: 支持/棄却]
  仮説2: [内容] → [検証結果: 支持/棄却]
  仮説3: [内容] → [検証結果: 支持/棄却]
  根本原因: [特定された原因]

■ 修正内容
  Level: [1/2/3]
  変更箇所: [ファイル名:セクション名]
  変更前: [概要]
  変更後: [概要]
  変更理由: [1文]

■ 効果測定
  測定期間: [修正後XX日間]
  修正前スコア: XX.X
  修正後スコア: XX.X
  改善幅: [+X.X / -X.X]
  判定: [成功/部分的成功/失敗/測定中]

■ 学び
  [この進化から得られた知見を1-2文で]
```

---

## 禁止事項
- スコアの水増し・自己評価の甘い採点（反証モードで自己バイアスを排除）
- 評価カードの省略（「簡単なタスクだから」は理由にならない）
- 修正なき低評価の放置（C判定以下は必ずアクションを取る）
- 効果測定なきスキル変更（変更前後の比較データ必須）
- 他エージェントの評価への干渉（自己評価の独立性を守る）
