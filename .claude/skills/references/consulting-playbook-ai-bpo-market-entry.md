## 1. ゲームチェンジ: 単価 → 数量 × 粗利率

従来の BPO は人海戦術で巨大化し、資本力ある大手が規模の経済で寡占した。AI は人件費と推論コストの圧縮を可能にし、従来不可能な粗利率を持つプレイヤーを生む（INFERENCE: JAFCO レポート フレーミング）。

処理単価は AI の普及とともに低下圧力が常に働くため「単価」勝負は構造的に不利。勝ち筋は「数量」（処理件数・顧客数の最大化）と「粗利率」（HITL 設計・推論コストマネジメント）の同時極大化である（INFERENCE）。

## 2. 2 参入機会の比較

| 軸 | (i) 大手寡占エンタープライズ向け BPO | (ii) AI で成立する新規 BPO（SMB 向け） |
|---|---|---|
| 市場構造 | 数千人 + オフショアで大手寡占 | 既存 BPO 事業者未参入・フラグメント |
| 参入障壁 | 大手並みキャパ + エンプラ内製化志向 / 既存システム接続要件の両立 | 低い（AI が処理単価を下げ SMB で初めて外注が経済的に成立） |
| 主な勝ち筋 | AI 常駐 + 人間エキスパートの HITL ハイブリッド、アライアンス / M&A でキャパ補完 | デファクト化スピード、AI-SDR で短リードタイム開拓、ワークフロー不変導入 |
| 営業 | 直販はセールスサイクル長期。ケイパ補完フックのアライアンスが有効 | AI-SDR で SMB を高速開拓 |
| ロックイン | 既存システム統合深度 | 処理件数 + データ蓄積 |
| 個人・小規模発の適合性 | 低（資本力 / 数千人体制が前提） | 高 |
| ConsultingOS 立ち位置 | アライアンス補完で部分関与（エンプラ自力参入は非適合） | 主軸（ICP §9.1 中以上の業界から選択） |

INFERENCE: 上記比較は JAFCO レポートのフレーミングを ConsultingOS 立場で再解釈したもの。

## 3. 事例: Crescendo（(i) エンタープライズ向け AI BPO）

- カテゴリ: CS 領域 AI BPO。完全自律型（Sierra / Decagon 等）と純粋人海戦術型の中間ポジション（FACT: JAFCO レポート記載）。
- 体制: AI 常駐 + 自社 CS エキスパートが例外処理・品質管理を担う HITL ハイブリッド（FACT: JAFCO レポート記載）。
- PartnerHero 買収: 体制（3,000-3,500 名、FACT: JAFCO レポート記載値）と顧客基盤（200 社以上、FACT: JAFCO レポート記載値）を早期構築するための M&A。体制・顧客を M&A で取得し、エンタープライズが求めるキャパと信頼実績をショートカット。
- 構造示唆: 個人・小規模発にとってエンプラ向け(i)の直接参入は資本力・キャパ要件で困難。ただし Crescendo 型企業とのアライアンス（ケイパ補完フック）で営業支援・専門領域補完に部分関与する二段構えは可能（INFERENCE）。

## 4. 事例: Tennr（(ii) SMB 向け新規 BPO）

- カテゴリ: 医療紹介状 / 患者情報処理 AI BPO（FACT: JAFCO レポート記載）。
- 業務フロー: FAX / メール書類を AI 読取り → 保険適用確認 → 事前承認 → 患者ルーティングを一気通貫処理（FACT: JAFCO レポート記載）。
- 専用モデル: RaeLM（医療書類処理専用モデル）を自社構築（FACT: JAFCO レポート記載）。

Tennr 成功要因の構造:

| 要因 | 内容 |
|---|---|
| デジタル化しない設計 | あえて FAX / メールのままで受け取る。現場の運用を変えないことで導入摩擦ゼロ |
| 専用モデル構築 | 汎用 LLM ではなく RaeLM で医療書類処理の精度と推論コストを最適化 |
| ROI の月次可視化 | コンバージョン率・返戻率・FTE 削減を月次で可視化し、支払い継続の正当化をクライアント側で完結させてロックイン |
| 件数ベース・成果報酬型 | 従来 BPO では採算が合わなかった小規模医療機関を顧客化 |

医療介護領域は件数ベース・成果報酬型の価格設定で、従来 BPO を利用できなかった小規模事業者を顧客化でき、潜在市場は既存 BPO を上回る可能性がある（SPECULATION: JAFCO レポート内の見立て）。

Tennr 型再現モデル（ConsultingOS への適用）: ワークフロー不変の導入 + 専用化（業界特化モデル or プロンプト）+ ROI 月次可視化 + データロックインの 4 ステップは、AI 派遣会社モデル（道 2）のデリバリー標準として使用できる（INFERENCE）。

## 5. 6 ポイント: コンサル部門 vs AI / サービス開発部門の分担

| ポイント | 主担当 | 内容 |
|---|---|---|
| マーケットアプローチ選定 | コンサル（competitive-analyst + strategy-lead） | 業務量・頻度・AI 代替適合性・既存寡占有無で(i)/(ii)を選定。ICP §9.1 Vertical AI 適合度と連動 |
| 特化・標準化設計 | 両部門交点（product-manager + ai-engineer） | 単価・粗利率とカスタマイズ工数のバランス。汎用 vs 専用モデル選択が設計の核 |
| 営業力 | コンサル（strategy-lead + proposal-writer） | (i) はアライアンス / M&A。(ii) は AI-SDR + 短リードタイム。sales-deck-designer と連携 |
| 粗利率（オペレーショナルエクセレンス） | AI / サービス開発部門（ai-engineer + infra-devops） | 人間と AI の役割分担設計（HITL 比率）、推論コストマネジメント（専用モデル vs 汎用 LLM コスト比較） |
| 顧客ロックイン | 両部門（strategy-lead + fullstack-dev） | (i) 既存システム統合深度。(ii) 処理件数 + データ蓄積 + ROI 可視化ダッシュボード |
| ファイナンス | コンサル（kpi-analytics + client-success） | リードタイム・1 人当たり生産性・受注予測・人材 / 販路 M&A（(i) 主）の検討 |

## 6. ConsultingOS 立ち位置

**主軸: (ii) SMB 向け新規 BPO**

理由:
1. 既存の「中小 / 個人 / OEM vertical 特化」方針（ExaWizards ベンチマーク等、consulting-playbook vertical OS クラスタ）と整合する。
2. Tennr 型（ワークフロー不変 + 専用化 + ROI 月次可視化 + データロックイン）が道 2（AI 派遣会社化）の再現モデルとして直接適用可能（INFERENCE）。
3. 個人・小規模発での参入が可能（資本力・大規模キャパ不要）。

**補完: (i) エンタープライズ向けはアライアンス**

ケイパ補完フックとしてのアライアンス（例: Crescendo 型企業との営業支援 / 専門領域補完）で部分関与する二段構えを採用。M&A は選択肢から外す（資本力要件から個人・小規模発に非適合）（INFERENCE）。
