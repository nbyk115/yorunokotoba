# ジャック・ドーシー流ミニ AGI 化企業 4 原則（2026-05-14 統合）

`strategy-lead` + `product-manager` + `client-success` + `proposal-writer` 連携の横断 4 原則。マスク 4 + アルトマン 7 + FDE 5 + Legora 5 + a16z Defensibility 7 軸と並列で、AI 中核化組織変革を扱う。

出典: ジャック・ドーシー（Twitter / Block 創業者）「あらゆる企業は今やミニ AGI になれる」(INFERENCE: ユーザー提示テキスト経由、一次出典 URL 未確認)。Block は 2025-02 に 4,000 人（40%）レイオフ実施、AI を既存組織への後付けでなく中核に据える賭けに転換（FACT: 公開報道、ただし本セッション URL 未取得）。

## 基本テーゼ（ドーシー本人発言）

- 階層構造は、多くの人間のあいだで情報を伝達するために存在する
- 企業のすべての成果物は、その会社の働き方 / 作り方 / 意思決定のシグナル
- すべての成果物の上にインテリジェンス層を載せると、会社そのもののモデルができあがる
- 今や誰でも、人の連鎖を待たずにリアルタイムで会社に問いかけられる
- 会社のアーキテクチャが、そのまま企業のスピードを決める
- 「コパイロット」捉え方は本質を外す（生産性改善でなく構造変化が起きている）
- 答え: インテリジェンスを中核に据えて会社を作り直すこと

## 原則 1: 3 役割集約モデル（IC / DRI / プレイヤーコーチ）

Block の組織改革で、役割は次の 3 つに集約:

| 役割 | 定義 | 人間に残る持続スキル |
|---|---|---|
| IC（ビルダー / オペレーター）| エージェントにより拡張、1 人で以前の 10 人分の仕事 | 判断力 / センス / 創造性 |
| DRI（Directly Responsible Individual）| 顧客成果 / 戦略 / ロードマップの責任 | オーナーシップ / 説明責任 |
| プレイヤーコーチ | 指示でなく一緒に手を動かしながら他者の能力を引き上げる | コーチング / 共感力 |

YOU MUST: 経営チームは 3 つすべてを担うことが求められる（特化型でなく統合型）。「マネージャー専業」役割は AI 中核化組織では構造的に不要、廃止対象。

ConsultingOS 整合:
- IC = 27 agent（1 人で 10 人分の物理化）
- DRI = assistant orchestrator（顧客成果オーナー、ユーザー = 真の DRI）
- プレイヤーコーチ = mid-turn correction + Strategy ⇄ Execution 循環（指示でなく一緒に手を動かす物理化）

ICP 提案質問 26 件目追加: 経営チームが「IC + DRI + プレイヤーコーチ」3 役割すべてを担っているか、マネージャー専業の中間管理職を廃止しているか（原則 1）。

## 原則 2: ロードマップ創発（顧客クエリ自動生成）

従来のロードマップは推測にすぎなかった。AI 中核化組織では:

- 顧客がまだ存在しない機能を求めると、システムが既存の能力からリアルタイムで構成
- システムが構成できなければ、そのギャップこそがロードマップになる
- 会話の中から自動的に浮かび上がる（インタビュー / サポートチケットから推測するものではない）
- 顧客が求めるものが自社のありたい姿と整合するかは最終的に人間が判断

YOU MUST: AI 導入提案で「ロードマップ生成方法」を診断軸化。インタビュー / サポチケから推測する従来手法を採用している企業は、AI 中核化に対応できない構造として警告。

ConsultingOS 整合:
- evolution-log = ユーザー指摘 + agent 失敗のリアルタイム捕捉（顧客クエリ自動収集の物理化）
- 反証チェック Step 4「リスク即潰し」+ システム構成不可能ケースを skill 化（ギャップ = ロードマップの物理化）
- 過去パターン: マスク 4 / アルトマン 7 / FDE 5 / Legora 5 / a16z Defensibility 7 すべて「ユーザー提示 → 即統合」フローで実装 = ロードマップ創発の実例

ICP 提案質問 27 件目追加: ロードマップが顧客クエリから自動生成される設計になっているか、インタビュー / サポチケ依存の従来手法に閉じていないか（原則 2）。

## 原則 3: 会議プロトタイプ持参（cost-of-trying ゼロ化）

会議のあり方が構造変化:

- 以前: スライド資料 / Google ドキュメント中心
- 今: 全員が実データ or 模擬データで動くプロトタイプを持参、その場でライブ修正
- 誤った方向を試すコストはほぼゼロ
- まず広く試し、その後に絞り込んでコミット

YOU MUST: B2B 提案で「会議の成果物形式」を診断軸化。スライド中心の意思決定プロセスは AI 中核化組織では構造的に陳腐化、プロトタイプ駆動への移行を推奨。

ConsultingOS 整合:
- visual v9 (HTML プロトタイプ) = 関根さん提案当日にライブ修正可能な物理化
- OEM ベースキット `oem-base-kits/n-y-craft-os/` = プロトタイプそのもの（125 ファイル即時取り込み可能）
- artifact / Claude Design / Playwright = cost-of-trying ゼロ化の AI 時代インフラ

ICP 提案質問 28 件目追加: 重要意思決定会議で「動くプロトタイプ」を持参する文化があるか、スライド / ドキュメント中心の従来形式に閉じていないか（原則 3）。

## 原則 4: 階層圧縮（5 層 → 2-3 層、構造変化）

Block の組織変革は階層圧縮の構造実証:

- 現在 CEO から IC まで 5 階層 → 今年中に 2-3 階層へ縮小目標
- 中間管理職は廃止、情報流通は AI が担う
- 4,000 人（40%）レイオフ = 圧縮の物理証拠（生産性改善でなく構造変化）
- 「1 年で大半の企業が同じ結論に至る」（ドーシー予測 SPECULATION）

YOU MUST: 組織設計提案で「現状階層数 → 目標階層数」を必須記載項目化。階層数を圧縮しない AI 導入は「コパイロット後付け」= 本質を外している兆候。

CEO の仕事の変化:
- 従来: 原則とチームを整え、顧客文脈に基づく意思決定を担保、実行水準を引き上げる
- 新しい: 「インテリジェンスとしての会社」を設計し、継続的に整合させ続ける

ConsultingOS 整合:
- ConsultingOS 自体が「階層ゼロ」の物理実装（ユーザー = 唯一の DRI、assistant orchestrator = フラット）
- 27 agent も階層なし（部門区分はあるが、起動順序はタスク駆動 = 創発的）
- ハードルール 17「主語詐称禁止」+ Autonomous Mode 既定化 = CEO の「新しい仕事」（インテリジェンスとしての会社の設計）の物理化

ICP 提案質問 29 件目追加: 組織階層数が AI 中核化に合わせて圧縮計画されているか、現状階層を維持したまま AI を後付けする「コパイロット型」に閉じていないか（原則 4）。
