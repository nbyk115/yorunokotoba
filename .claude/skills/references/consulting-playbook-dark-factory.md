## 1. ダーク・ファクトリーとは

製造業発想からの転用:
- 工場が完全自動化で人がいなくなる → 照明全部消せる（人がいないなら機械は真っ暗でも動く）
- これをソフトウェア開発に適用: 「人間はコードを一切レビューしない、中身も見ない、それでもプロ品質基準を満たす」

バイブ・コーディング（投げっぱなしで何が出てくるか分からない）との違い:
- バイブ: 品質基準なし / 偶然依存
- ダーク・ファクトリー: 品質基準は押さえたまま + コード自体は直接見ない（構造的に保証）

## 2. 原則 1: 人間レビュー除去 + プロ品質維持の両立構造

YOU MUST: AI 駆動開発提案で「人間レビューが品質ボトルネック」と認識し、レビュー除去 + 品質維持の両立を設計目標化。「全部 AI に任せる」or「人間が全レビュー」の二極対立でなく、構造的に品質を保証する第三の道。

ConsultingOS 整合（前 8 体系との接続）:
- アルトマン式原則 1 AI 1 人チーム: ダーク・ファクトリー = チーム不要の物理実装
- アルトマン式原則 8 アイデアマンの逆襲: 非テクニカル創業者でもダーク・ファクトリー運用で品質ソフトウェア構築可能
- マスク式原則 3 Unkind Truth: 「人間レビュー必須」前提を直視して捨てる
- FDE 時代戦略原則 3 業務インフラ再設計: QA / レビュープロセスの根本再設計
- a16z Defensibility「Recreation risk」: 「最初の 80% は AI で再現容易、エッジケース防御線が moat」とダーク・ファクトリーは同型

## 3. 原則 2: Agent 群れ QA シミュレーション（StrongDM 実例）

StrongDM 実装パターン:
- エンドユーザー業務環境をまるごとシミュレーション（シミュレートされた Slack チャンネル + シミュレートされた従業員群）
- 「ねえ、誰か Jira のアクセス権くれない？」みたいな会話を 24 時間延々と投げ続ける
- 月 1 万ドル規模のコスト → 「決して眠らない QA 部門」
- 結果: あらゆるパターンでガンガン叩かれ、頑丈にテストされる
- 製品領域: アクセス管理セキュリティ SaaS（「ノリだけでコード書いていいジャンルじゃない」と思われる重い領域）= ダーク・ファクトリー適用の高難度実証

YOU MUST: AI 駆動開発提案で「Agent 群れによる業務シミュレーション QA」を Phase 2 オプションとして提示。エンドユーザー / 業務環境のまるごとシミュレーション設計が次世代 QA の標準。

ConsultingOS 整合:
- a16z Defensibility「Action exhaust」: Agent 群れ実行ログ = proprietary data 化、まさに行動ログ資産
- ドーシー流原則 3 会議プロトタイプ持参: cost-of-trying ゼロ化のソフトウェア版（月 1 万ドル = プロトタイプ 1 個より安い 24/7 QA）
- ハードルール 17 Autonomous Mode 既定化: ConsultingOS 自体が agent 群れ運用、反証チェック + 規律 hook = 自動 QA

## 4. 原則 3: 責任設計の未解決問題（structurally open）

サイモン・ウィリソン提起:「コードレビューしないで、どうやってソフトウェアの品質を確かめるか?」「どうすればそのやり方で責任を持ってソフトウェアを作れるか?」

= 未解決問題。ダーク・ファクトリーが普及するには「責任の所在」設計が必要:
- 誰が最終責任を負うか（CEO / DRI / agent 設計者）
- 障害時の説明責任（agent ログ + 設計意図 + シミュレーション結果の連結）
- 規制対応（金融 / 医療 / 製薬 vertical では特に重要）

YOU MUST: ダーク・ファクトリー型提案で「責任設計」を必須記載項目化。責任所在不明な提案は規制 vertical では却下、汎用 vertical では Phase 2 で責任設計フェーズを組み込み。

ConsultingOS 整合:
- ドーシー流原則 1 3 役割集約 DRI: ダーク・ファクトリーの責任所在 = DRI 集中型キャリアの実例
- アルトマン式原則 2 エージェント管理 CEO スキル: agent 群れ運用の責任を CEO が負う前提
- ハードルール 17 主語詐称禁止 + Autonomous Mode: agent 起動主体の明示 = 責任設計の物理化
