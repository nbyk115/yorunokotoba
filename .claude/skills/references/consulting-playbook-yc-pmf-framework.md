# Y Combinator PMF (プロダクトマーケットフィット) 5 フレーム（2026-05-14 統合）

`strategy-lead` + `product-manager` + `proposal-writer` + `client-success` + `feedback-synthesizer` 連携の PMF 探索 / 判断 / 加速フレーム。マスク 4 + アルトマン 10 + LayerX Be Animal + PoC 地獄脱出 4 戦略と並列で、PMF プロセス全体を体系化。

出典: Y Combinator「プロダクトマーケットフィットの見つけ方」(INFERENCE: ユーザー提示画像経由、Weebly 事例ベース)。

## 1. 核心: PMF は一夜にしてならず (Weebly 12 年事例)

Weebly = 50M ユーザー / 365M 売却まで 12 年。PMF 後の成長は急激だが、そこまでが長い旅。

YOU MUST: PMF 探索期間 = 平均 2 年以上の前提受容。「諦めるタイミングを間違えるな」。短期 metrics で PMF 判断するのは構造的誤り。

ConsultingOS 整合:
- LayerX Be Animal (PR #152) Case 3 祖業強迫観念回避 = PMF 探索期間中の固執リスク警告
- PoC 地獄脱出 4 戦略 (PR #161) = PMF 前後の戦略選択

## 2. フレーム 1: PMF 判断基準 5 シグナル

| シグナル | 説明 | ConsultingOS 適用 |
|---|---|---|
| 引っ張られる感覚 | ユーザーが自分でプロダクトを広め始めた | 関根さん N&Y Craft = 既に「OEM 化したい」発言 = 部分的 PMF シグナル |
| 解約率低下 | 使い続けるユーザーが増えた | 月次顧問契約継続率で測定 |
| サポートリクエスト変化 | 「壊れた」→「もっとこう使いたい」 | 関根さん Phase 1 で要望リクエスト記録 |
| 競合からの戻り | 「やっぱりこっちがいい」 | OEM 比較で確認 |
| 「もしなくなったら困る？」40% Yes | 強い PMF シグナル | 関根さん / 水野さんに直接ヒアリング推奨 |

YOU MUST: PMF 判断時に 5 シグナルを定量測定、感覚判断禁止。

## 3. フレーム 2: ユーザーインタビュー実践 5 原則

1. 解約したユーザーに電話せよ (最も正直なフィードバック)
2. 「何が嫌だったか」より「どんな問題を解決しようとしていたか」 = Jobs to be Done 理解
3. ユーザーの言葉をそのままプロダクトに反映するな (背景にあるニーズを汲む)
4. 週 10 件以上のインタビューを PMF 前は続けろ (量が質を生む)
5. 「代替品は何か」必ず聞け (競合は予想と違う)

ConsultingOS 整合:
- feedback-synthesizer agent (PR #DH) = Jobs to be Done 分析の物理化
- 関根さん / 水野さんの「解約ユーザー」は現状なし、潜在顧客 (他クラフトビール業者 / 他投資先候補) ヒアリング候補

## 4. フレーム 3: ピボット判断 5 原則

1. ピボット = 失敗でなく「学習の実装」(Weebly も初期コンセプトから変わった)
2. コアチームの信頼関係を保ちながらピボット
3. 「市場が間違ってる」vs「プロダクトが間違ってる」を区別 (前者諦め、後者直し)
4. 小さなピボット (機能レベル) を繰り返すのがベスト、大きなピボットは消耗
5. ユーザーが予想外の使い方を発見 → それがピボット方向

ConsultingOS 整合:
- マスク式原則 4 アルゴリズム実行 5 ステップ「サイクル加速」と整合
- アルトマン式原則 9 現実主義スタンス = 諦観 / 狂気の中道でピボット判断
- 関根さん N&Y Craft = ConsultingOS の vertical AI-OS ピボット候補の実例 (コンサル → vertical SaaS)

## 5. フレーム 4: PMF 後の成長加速 5 原則

1. 「一つのことが誰よりもうまい」(全機能 2 番 < 1 機能 1 番)
2. PMF 後はスピードがすべて (遅い成長 = 競合追いつく)
3. 配布チャネルを早期に確立 (プロダクトと同じくらい Go-to-Market 投資)
4. チームサイズを PMF 段階に合わせる (PMF 前の大人数採用は組織歪める)
5. 半数の米国民が Weebly サイトを月 1 訪問 = ユーザーの起業家たちの成功の証

ConsultingOS 整合:
- アルトマン式原則 3 垂直特化 = 「一つのことが誰よりもうまい」と同型
- ドーシー流原則 4 階層圧縮 = PMF 前の大人数採用回避
- Legora 流原則 2 超高速スケーリング = PMF 後スピード加速

## 6. フレーム 5: 50M ユーザー → 365M 売却 = 12 年の旅

Weebly タイムライン (FACT、YC 提示):
- 最初のコードから 6 ヶ月後もまだローンチしてなかった
- YC 応募締め切り 1 時間前にチャンス到来、準備より行動
- 最初の TechCrunch 掲載スパイクあっても次の日には元に戻る (プレスは PMF の代替にならない)

ConsultingOS 整合: アルトマン式原則 9 現実主義 = 「3-5 年でシンギュラリティ」極端論でなく 12 年スパンの現実主義中道。
