# Superpowers — ソフトウェア開発コアスキル集

## 概要
Jesse Vincent（obra）が開発した、ソフトウェア開発の全SDLCをカバーするコアスキル集。
TDD・体系的デバッグ・実装前計画・完了前検証・サブエージェント協働の5本柱。
出典: obra/superpowers (v5.0.6)

---

## §1 Writing Plans — 実装計画作成

### 原則
「コードを書く前に、必ず詳細な計画を書く。計画なしの実装開始は禁止。」

### 計画書フォーマット
```
保存先: docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md

## ファイル構造
[新規作成・変更するファイルと役割を先にマップ]

## タスク一覧
### Task 1: [タスク名]
**変更ファイル**: [正確なファイルパス]
**ステップ**:
1. [実際のコード付き 2-5分のステップ]
2. [プレースホルダー（TBD/add validation等）禁止]
3. 検証: [実際のコマンドと期待出力]
**コミット**: [コミットメッセージ]
```

### 品質基準
- 各タスク: 2〜5分で完了できる粒度
- 全コードブロック: 実際のコードを記載（プレースホルダー禁止）
- 型・関数名・プロパティ: タスク間で一貫性を保つ
- 参照する関数/型: 必ず先のタスクで定義済みにする
- YAGNI・DRY・TDD原則を遵守

### スコープ確認
- 独立したサブシステムにまたがる場合は計画を分割する
- 計画完成後、実行方法を選択:
  - **Subagent-Driven**: タスクごとに新しいエージェントを投入
  - **Inline Execution**: バッチ実行（同セッション内）

## §2 Systematic Debugging — 体系的デバッグ

### 鉄則
「修正前に必ず根本原因を特定せよ。根本原因なしのFIXは禁止。」

### 4フェーズプロセス

**Phase 1: 根本原因調査**
```
- エラーメッセージを注意深く読む
- 問題を一貫して再現させる
- 最後に正常動作した時点を特定
- コンポーネント境界でデータフローを逆トレース
- 診断証拠を収集
```

**Phase 2: パターン分析**
```
- 動作する類似コードを探す
- リファレンス実装を完全に読む
- 具体的な差異を特定
- 全依存関係を把握
```

**Phase 3: 仮説と検証**
```
- 明確な仮説を1つ立てる
- 最小限の変更でテスト（一度に1変数）
- 結果を確認してから次へ
```

**Phase 4: 実装**
```
- 失敗するテストを先に書く
- 根本原因を解決する1つの修正を実装
- 解決を確認する
```

### 構造的問題の判定
**3回修正しても新たな問題が現れる場合 → パッチ適用を停止し、設計見直しを議論する**

### 禁止事項
- データをトレースせずに解決策を提案
- 複数の修正を同時に試みる
- テストをスキップ
- 「急いでいる」「シンプルだから」を理由に省略

## §3 Verification Before Completion — 完了前検証

### 鉄則
「完了・修正・テスト通過を主張する前に、必ず新鮮な検証証拠を実行すること。」

### 5ステップゲート
```
1. Identify: 自分の主張を証明するコマンドを特定する
2. Execute: 新たに・完全に実行する
3. Read: 完全な出力とexit codeを読む
4. Verify: 結果が自分の主張と一致するか確認する
5. Claim: 証拠を添付してのみ主張する
```

### 「完了」とみなされないもの
- 「動くはずです」（テストなし）
- 部分的チェック（Linter通過 ≠ コンパイル成功）
- 前回のテスト実行結果
- エージェントの成功レポートを独立確認なしに信頼
- 検証前の満足表明

## §4 Test-Driven Development — テスト駆動開発

### 鉄則
「テストなしに本番コードを1行も書かない。」

### RED-GREEN-REFACTORサイクル
```
RED: 失敗するテストを1つ書く → 失敗を確認（期待通りの失敗メッセージを確認）
GREEN: テストを通過する最小限のコードを実装
REFACTOR: テストをグリーンに保ちながらコードを改善
```

### 非交渉ルール
- テスト前に書いた本番コードは削除する
- 「リファレンス」実装をテスト横に残さない
- 手動テストは自動検証の代替にならない
- 「シンプルすぎてテスト不要」は拒否する

### なぜテストファーストか
- 実装後のテストは即座にパスし、実際の問題を検出しない
- 先にテストを書くことで、API設計とエッジケースを実装バイアスなしに考えられる

## §5 Subagent-Driven Development — サブエージェント協働開発

### 適用条件
- 完全な実装計画がある
- 独立したタスクが大半
- 現セッションに留まる必要がある

### プロセス
```
1. 計画から全タスクを必要なコンテキスト付きで抽出
2. タスクごとに: 実装エージェント投入 → Q&A → 実装 → 仕様レビュー → コード品質レビュー
3. フィードバックループ: 問題発見 → 修正 → 再レビュー
4. 全タスク完了後: 最終コードレビュー → ブランチ完了
```

### モデル選定
- 機械的な実装タスク（独立関数・明確な仕様・1-2ファイル）: 高速・低コストモデル
- 設計判断・広いコードベース理解が必要: 高性能モデル

### 禁止
- レビューをスキップ
- 問題未解決のまま次タスクへ進む
- ブロックしたサブエージェントに同じ指示で再試行
