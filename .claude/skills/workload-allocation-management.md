# Workload Allocation Management: 業務負荷配分マネジメント

## このスキルの存在意義

「人を潰す組織は、業務配分を数値管理できていない」（2026-05-07 ConsultingOS 統合・PR AX）。

マネージャーの役割は「仕事を振ること」ではなく「負荷と成長の設計」。優秀なエースに業務集中する組織は育成放棄の証拠。「忙しそうな人に振る」は思考停止。本質は「誰に依存している状態か」を放置していること、仕組みの問題として捉え直さない限り偏りは再生産される。

ConsultingOS は 27 エージェント体制で同じ構造リスクを抱える。本 skill は自己適用 + 外販ソリューション + 教育実習の 3 形態で物理化する。

## 1. 配分スコアリング（複数軸、Goodhart 対策必須）

### 1.1 基本スコア（難易度 × 件数）
```
案件スコア = 難易度（1-5） × 件数 × 関与時間係数
個人合計 = Σ 案件スコア（直近 4 週間）
```

難易度判定基準:
- 1: ルーチン・テンプレ流用
- 2: 一部カスタマイズ
- 3: 標準的な分析・提案
- 4: 高度な構造分析・新規フレーム必要
- 5: 経営判断級・前例なし

関与時間係数: フルアサイン = 1.0、レビュアー = 0.3、サポート = 0.5

### 1.2 育成効果スコア（必須併設、単一指標最大化対策）
```
育成スコア = 直前期スコア成長率 + 高難易度（4-5）案件比率
```
低難易度ばかりやっている = 数値は上がるが育成ゼロ = Goodhart 該当の典型。

### 1.3 反指標（必須併設、Goodhart 反証）
- バーンアウト指標: 残業時間 / 連続稼働週数 / 1on1 ネガ発言頻度
- 単一依存指標: 「この人がいないと回らない」案件率
- チャーン早期警戒: 退職リスクスコア（HR フラグ + 上司面談記録）

### 1.4 定性レビュー（数値だけで判断しない構造担保）
4 週間ごとに以下を必ず実施:
- スコアと実感のギャップヒアリング（本人 + 上司 + 同僚）
- 「忙しい ≠ 成果出している」を識別する定性チェック
- 数値が捉えきれていない属人性 / 信頼関係 / モチベーション質的要素の評価

## 2. 偏り検知ルール（自動アラート）

### 2.1 個人集中アラート
- 個人合計スコアが「チーム中央値 × 1.5」を超過
- かつ反指標（バーンアウト / 単一依存）が悪化中

### 2.2 育成放棄アラート
- 個人の育成スコア（高難易度比率）が「チーム平均 × 0.5」未満が 8 週間継続
- かつ個人合計スコアは平均以上

### 2.3 単一依存アラート
- 特定エージェント / 特定メンバーへの依存率（全案件中の関与率）が 40% 超

## 3. 再配分プロトコル

検知後 1 週間以内に必ず実行（放置は仕組み違反として記録）:

1. 該当者と上司で「降ろすべき案件」を決定（量より質、低難易度優先で剥がす）
2. 受け手候補を育成スコア低位から指名（育成機会として配分）
3. 引き継ぎ期間 2 週間（属人化対策として並走）
4. 4 週間後に再評価（偏りが再発していないか / 受け手のバーンアウト警戒）

## 4. ConsultingOS 自己適用（hook 物理化）

### 4.1 workload-tracking.sh
- SessionStart hook で 27 エージェント別起動回数を直近 4 週間集計
- 「特定エージェントへの集中 = 単一依存」を検出
- 関根さん案件型の sales-deck-designer 過集中も検出可能
- 集計結果: `.claude/memory/workload-snapshot.json` に保存、4 週間分保持

### 4.2 偏り検知時の挙動
- assistant 環境変数 `CONSULTINGOS_WORKLOAD_WARNING` に検知メッセージを注入
- 次セッション開始時に「特定エージェントへの集中検出: 他エージェントの並列起動を検討してください」を提示

### 4.3 評価期日
- 4 週間ごとに偏り検知ロジック自体を再評価（Goodhart 対策）
- 単一指標化していないか / 反指標が機能しているか
- 形骸化していれば skill / hook 改修

## 5. 外販ソリューション（PSA 差別化）

既存 PSA ツール（FunctionFox / Resource Guru / Kantata 等）との差別化軸:

### 5.1 差別化 1: 数値 + 定性ハイブリッド
PSA は数値最適化のみ、ConsultingOS は定性レビュー併用を構造化。

### 5.2 差別化 2: 育成効果スコア併設
PSA は配分効率指標、ConsultingOS は育成スコア + 反指標で消耗を防ぐ。

### 5.3 差別化 3: 4 週間再評価ルール
スコアロジック自体の形骸化を構造的に潰す（Goodhart 法則対策）。

### 5.4 差別化 4: AI orchestrator 内蔵
配分最適化を自然言語で対話可能（Claude Code ベース）。

### 5.5 価格モデル（INFERENCE、コンサル系 SaaS から推定）
- 月額 30-100 万円 / 社（中堅コンサル / 制作会社 / SaaS 開発組織向け）
- 初期診断 100-300 万円
- 教育実習プログラム 50-200 万円 / 回
- ROI: 1 件「エース消耗による退職」防止で数千万円損失回避

## 6. 教育実習プログラム

### 6.1 月次社内ワークショップ（90 分構成）
- 0-15 分: 「忙しそうな人に振る」を実例で再現（ロールプレイ）
- 15-45 分: 自社の偏り実態を本 skill のスコアで可視化
- 45-75 分: 再配分プロトコル演習（実案件 1 件）
- 75-90 分: 4 週間後の再評価コミットメント宣言

### 6.2 対象組織
- コンサルティングファーム（マネージャー以上）
- 広告代理店 / 制作会社（PM / プロデューサー）
- SaaS 開発組織（EM / TL）
- ConsultingOS 自体（orchestrator 規律として 4 週間ごと自己適用）

### 6.3 認定制度（オプション）
- 4 週間運用 + 偏り検知 + 再配分実施で「Workload Allocation Manager」認定
- 認定者は ConsultingOS の skill 改善提案権限を保有

## 7. 連携先 agent

- `strategy-lead`: 人材アロケーション診断 S6 シナリオで活用
- `product-manager`: リソース管理プロセス設計
- `client-success`: クライアント側のヘルススコア（消耗 = チャーン予兆）連動
- `kpi-analytics`: スコア集計ロジック / 反指標設計
- `tech-lead`: hook 実装 + workload-snapshot.json 管理

## 8. 形骸化検知

本 skill は 4 週間ごとに再評価:
- スコアロジックが Goodhart 法則化していないか
- 反指標 / 定性レビューが運用されているか
- 教育実習が「やった」だけになっていないか

形骸化検知は `evolution-log.md` に記録、Boris #3 ruthlessly edit で項目削除 / 統合判断。
