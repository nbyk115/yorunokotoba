# sales-deck-review

使用タイミング：B2Bセールス資料（特に日本語・代理店提案）のレビューやリライトを求められたとき。

## 優先チェックリスト（上から順に実行）

### 1. コピーの具体度

- タグライン／ヘッドラインが **主語 × 動詞 × 目的** で完結しているか
- 抽象語（「気づき」「体験」「共感」）を定義なしで使っていないか
- **上から目線**（「〜できます」「〜組めます」「代理店が」）を検出 → 「〜いただけます」「代理店様」に言い換え
- 無根拠の最上級（「最も」「業界最大級」「唯一」）→ 実証可能な表現に置換

### 2. ハルシネーション検出

- すべての数値にソースURL + 発行日が紐づいているか
- 派生数値（「80%が…」など）が一次ソースから導けるか
- 競合情報が公式／IR／主要紙ソースか
- 代理店ヒアリング等の**非公開データを資料に載せていない**か

### 3. 競合比較表

- 「非公表 / 要相談」が半数以上の行は **削除**
- 自社が負ける軸は載せない or 軸名を変えて文脈化
- 自社の弱みを **自分で surface しない**（留意点行なし）
- 商品名（MTA / GOAT / AnyTag 等）だけでなく **解釈した強み** を主文に
- 競合カテゴリによって領域を限定（例：総合広告グループなら "インフルエンサー領域での" 強みに限定）

### 4. レイアウト違和感

- **横揃え**：複数カードの proof box / KPI / CTA が横一線にあるか
- **縦揃え**：下段（.srcLine / .caseNote / .footnote）が下端に pin されているか
- **改行**：日本語が文節境界で改行されているか。`word-break: auto-phrase; line-break: strict;` が設定されているか
- **文字切れ**：`transform: scale()` や `bottom: -x` で突き抜けた要素がないか
- **重複情報**：見出しに含まれる数字を直下の大数字で繰り返していないか

### 4.1 並列スライド同型構造チェック（B4・関根さん案件 v14 学習・2026-05-07 物理化）

連番スライド（例: ストーリー 1/2/3, ケース A/B/C, Phase 1/2/3）では:

- [ ] 同一テンプレート構造を強制（`.card.before` / `.card.after` 等）
- [ ] card 末尾の footnote パターン（`.small accent` 等）は全スライドで揃える
- [ ] blockquote / 黒ボックス等の追加要素は全スライドに揃えるか全削除
- [ ] grid layout 数（`.grid-2` / `.grid-3`）を統一

判定: 連続スライドの 1 枚だけ末尾構造が違う場合（v14 で発生: ストーリー 3 だけ下部 blockquote 帯あり）、**Sato 流 ruthlessly edit に従い削除側で統一**。追加で揃えると情報冗長。

検証: `grep -A3 "ストーリー [0-9]" *.slides.md` 等で並列スライドの末尾構造を目視 diff。

### 4.2 ブロック密度制限（B5・関根さん案件 v19 学習・2026-05-07 物理化）

1 列内に視覚的に独立した装飾ブロック（blockquote / 黒ボックス / accent box / card 等）を:

- 推奨: 1 列に 1-2 ブロック（max 2）
- 警告: 3 ブロック以上（ガチャガチャ要因、視覚的混雑）
- 禁止: 4 ブロック以上

統合方法: 「結論」と「補足」は同一ブロック内で階層化（caption + main + sub）

検証: `.grid-2 > div` 内の独立装飾要素を grep カウント。

### 5. 情報設計

- 1ページ 1メッセージになっているか
- 自社紹介より先に売り込みを始めていないか（P2=会社紹介、P5以降=ポジショニング）
- 各ページの役割（静→知→強→信→実務→決）が通っているか

### 5.1 連続スライドの relationship 明示（B9・関根さん案件 v18 学習・2026-05-07 物理化）

ベース vs カスタム、戦略 vs 実装等の対応関係があるスライドペアでは、冒頭/末尾に必ず relationship を明示:

- ベース説明スライド: 「これは標準装備、次のページで [カスタム箇所] を説明」
- カスタムスライド: 「(前のページの) ベースに対して以下を上書き・追加」
- 戦略 → 実装: 「戦略 (P5) を実装に落とすと P10-15 の構造になる」

判定: ページ単独で読むと意味が伝わらない・前後を見ないと不明なスライドが連続する場合、relationship 注釈を必ず追加。

### 6. クライアントリテラシー対応（B1・関根さん案件 v6/v14/v15 学習・2026-05-07 物理化）

業界略語・専門用語の初出注釈確認:

- [ ] PMF / CAGR / Multi-Agent / RAG / LP / KPI / ICP / DESIGN / LLM / Fine-tuning 等の AI / ビジネス略語を全ページ grep
- [ ] 初出に括弧展開がない箇所は REJECT
- [ ] 同義の日本語が一般的な用語は略語ではなく日本語で表記（例: ICP.md → 顧客像定義書）
- [ ] AI 関連用語（ChatGPT / Claude / NotebookLM 以外）は AI 素人クライアント向けに必ず注釈

検証: `grep -nE "PMF|CAGR|Multi-Agent|RAG|LP[^a-z]|KPI|ICP|DESIGN|LLM|fine-tuning"` で出現箇所を抽出、初出に注釈があるか目視確認。

## 重要原則

1. **敵地で戦わない**。自社が負ける土俵には乗らない
2. **断言 > 婉曲**。曖昧な敬語で戦闘力が落ちる
3. **数字は必ず文脈化**（「25,000名の "稼働" 数」vs 相手の「400万 "累計" DB」）
4. **実名 vs 匿名** の判断：一次公開ソースあれば実名、なければ匿名＋口頭共有

## 出力フォーマット

レビュー時は下記の形で返す：

```
## ❌ 要修正（Critical）
- [ページ/要素]: 問題 → 推奨修正

## ⚠️ 検討（Warning）
- [ページ/要素]: 気になる点 → 代替案

## ✅ 維持（Keep）
- 特筆すべき良い点
```

リライト時は、修正前後の比較を必ず見せる。
