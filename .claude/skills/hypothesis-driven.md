# Hypothesis-Driven Problem Solving — 仮説ドリブン問題解決スキル

## 概要
McKinsey流「答えから逆算する」思考法。問いに対して最初に仮説を立て、それを証明・反証することで、最短経路で真の問題と解に到達する。
「調べてから考える」ではなく「考えてから調べる（検証する）」が核心。

---

## §1 起動トリガー

以下のいずれかに該当する場合に本スキルを適用する:
- 「何が本当の問題か分からない」「問題の構造を整理したい」
- 「どこから調査すればいいか」「調査方針を決めたい」
- 複雑な意思決定・参入判断・施策設計の前段
- 「なぜ売上が下がっているか」「なぜユーザーが離脱するか」の診断

---

## §2 仮説思考の基本原則

```
❌ データ収集型（典型的な失敗）:
  問い → 全データ収集 → 分析 → 結論 → 行動
  問題: 時間がかかる / 無関係データで埋まる / 結論が曖昧になる

✅ 仮説ドリブン型（McKinsey流）:
  問い → 仮説 → 最小限の証拠収集 → 検証 → 修正 or 確定 → 行動
  利点: 速い / 焦点が絞れる / 外れた仮説から即座に学べる
```

### 黄金律
1. **仮説は「答え」の形で書く**: 「問題はXだ」「解はYだ」（「Xが問題かもしれない」は仮説ではなく疑問）
2. **仮説は反証可能でなければならない**: 「どうすれば間違いだと証明できるか」を先に定義する
3. **1つの問いに1つの核心仮説**: 分散禁止。仮説は1文で言える最も重要なものを選ぶ

---

## §3 問題の構造化（Issue Tree）

### MECE分解の手順
```
Step 1: 核心問いを1文で定義
  例: 「なぜQ3の売上が前年比-20%か？」

Step 2: Level 1分解（MECE・3〜5枝）
  売上 = 顧客数 × ARPU
  → 顧客数が減ったか？ / ARPUが下がったか？

Step 3: Level 2分解（各枝を再度MECE分解）
  顧客数 = 新規 + 継続 - 解約
  → 新規獲得が落ちたか / 解約が増えたか

Step 4: 最もインパクトの大きい枝に仮説を配置
```

### Issue Treeの品質チェック
- [ ] MECE（漏れなく・重なりなく）か？
- [ ] 各枝は「検証可能」か？（データ or 観察で確認できる）
- [ ] 最も重要な枝を優先しているか？

---

## §4 仮説の立て方（核心仮説の設計）

### 仮説の3レイヤー構造
```
Layer 1: 状況仮説（Situation）
  「現在何が起きているか」の診断
  例: 「解約率が上昇しているのは、オンボーディングの離脱が原因だ」

Layer 2: 原因仮説（Complication）
  「なぜそれが起きているか」の解釈
  例: 「初回ログイン後72時間以内にコアアクションが完了しないユーザーが解約する」

Layer 3: 解決仮説（Resolution）
  「何をすれば解決できるか」の提案
  例: 「オンボーディングにプッシュ通知を3回追加すれば解約率を-15%改善できる」
```

### 仮説の強さ評価基準
| 基準 | 弱い仮説 | 強い仮説 |
|---|---|---|
| 具体性 | 「品質が問題だ」 | 「P99レイテンシが500ms超のセッションの解約率が2倍高い」 |
| 数値化 | 「改善できる」 | 「CVRを+20%改善できる」 |
| 反証条件 | 定義なし | 「A/Bテストでリフトが5%未満なら棄却」 |
| アクション | 曖昧 | 「具体的な施策Xを実施する」 |

---

## §5 検証プロセス（最速ファルシフィケーション）

### ファルシフィケーション優先原則
「仮説を証明する証拠」より「仮説を反証する証拠」を先に探す。
反証に失敗した仮説の方が信頼度が高い。

```
検証ステップ:
1. Critical Test（致命的テスト）の特定
   → 「これが間違っていたら仮説全体が崩れる証拠は何か」

2. 最小コスト検証の設計
   → 既存データで確認できるか？ → まずデータ
   → ユーザーインタビュー5人で確認できるか？ → 次に定性
   → A/Bテストが必要か？ → 最後に実験

3. 検証結果の分類
   ✅ 支持する証拠: 仮説の信頼度UP
   🔴 反証する証拠: 仮説を修正または棄却
   ⚠️  中立の証拠: 次の検証設計へ
```

### 検証の速度階層
| 階層 | 方法 | 時間 | コスト |
|---|---|---|---|
| 1st | 既存データ分析 | 数時間 | ゼロ |
| 2nd | ユーザーインタビュー（5人） | 1-2日 | 小 |
| 3rd | ランディングページMVP | 1週間 | 中 |
| 4th | A/Bテスト | 2-4週間 | 中〜大 |
| 5th | 本格実装 | 数ヶ月 | 大 |

**鉄則**: 下の階層に行く前に上の階層で棄却できないか確認する

## §6 ピラミッドストラクチャー（アウトプット設計）

仮説検証の結果を「答え先行」で伝える。

```
ピラミッドの構造:
頂点（結論）: 「問いへの答え」を1文で
   ↓
Level 1（理由）: 結論を支える3つの根拠（MECE）
   ↓
Level 2（証拠）: 各根拠を支えるデータ・事例

例:
「Q3売上-20%の主因は新規顧客獲得の停止だ（結論）」
  → 根拠1: 継続顧客のARPUは前年比+5%（維持されている）
  → 根拠2: 新規顧客数が前年比-40%（獲得停止の証拠）
  → 根拠3: 解約率は前年と同等（定着はOK）
```

### ソートウィズダム（So What？）テスト
各スライド・各段落に「So What?」を問う:
- 「このデータは何を意味するか？」の答えが明示されているか？
- 「だから何をすべきか」がアクションとして導けるか？

## §7 ConsultingOS適用

### 適用タイミング
| 状況 | 使い方 |
|---|---|
| 案件受付直後 | 核心問いを定義 → Level 1仮説を3つ立てる → インテークに反映 |
| 競合分析前 | 「競合に負けている仮説」を先に立ててから deep-research に引き渡す |
| KPI分析 | PL悪化の原因仮説を先に → kpi-analytics が検証データを収集 |
| 提案書作成 | ピラミッドストラクチャーで構成 → proposal-writer に引き渡す |
| 戦略判断 | 「Go仮説」を先に立て、反証を探す → strategy-lead の判断枠組みに |

### エージェント連携
```
hypothesis-driven（本スキル）の出力:
  → 核心仮説 × 3
  → Critical Test（致命的テスト）の定義
  → 検証優先順位（Layer 1〜5）
    ↓
deep-research または kpi-analytics が検証データを収集・分析
    ↓
strategy-lead または proposal-writer が最終判断・成果物化
```

### 反証モードとの統合

仮説の立て方はhypothesis-drivenが担当し、仮説の検証・反証はCLAUDE.md「反証モード」のトリプルチェックで実施する。

```
Step 1（自己反証）← 仮説を立てた時点で「この仮説が間違いの場合は？」を問う
Step 2（構造反証）← Issue TreeのMECE性・数値根拠の妥当性をチェック
Step 3（実用反証）← 「この仮説に基づく提案を実行したら何が起きるか」
```

## §9 品質基準

| 基準 | 内容 |
|---|---|
| 仮説の具体性 | 数値・条件・対象が明確か |
| 反証可能性 | 「どうすれば間違いだと証明できるか」が定義されているか |
| 最速検証 | より低コストの検証方法を先に使い切っているか |
| ピラミッド整合 | 結論→根拠→証拠が一貫しているか |
| So What？ | すべての分析に「だから何か」が付いているか |

## 🔺 反証モード（トリプルチェック必須）

本スキルを適用する全アウトプットは、CLAUDE.md「反証モード」セクションに定義されたトリプルチェックを必ず実行すること。

1. **自己反証**: 立てた仮説への反論・反例を最低3つ挙げ、確証バイアスを排除
2. **構造反証**: Issue TreeのMECE性・仮説の論理的整合性・抜け漏れを検証
3. **実用反証**: 検証コスト・実行可能性・ビジネスインパクトを検証

チェックなしのアウトプットはドラフト扱い。省略禁止。
