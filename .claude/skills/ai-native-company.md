# AI Native Company — AI Agent Company 設計原則

## 概要

佐藤裕介（FreakOut→STORES→AI事業）× 小野寺信行（電通デジタル）の統合知見を、
**2025年以降のAI Agent Company設計原則**として体系化したスキルファイル。

**核心命題:**
> 従来の「人が主体、AIがツール」ではなく「AIが主体、人が判断・関係・例外処理」の会社構造。
> 競合が人件費で動いている間、AIで動く事業は構造的に有利。

## 1. AI Agent Company の定義と判断基準

### 定義

```
AI Agent Company（AI Native会社）の3条件:

1. 【収益生成の主体がAI】
   売上を生む価値提供の60%以上をAIエージェントが担う
   （人が確認・承認するが、生産はAI）

2. 【変動費がAPI費用】
   売上増加に伴う変動費の主体が人件費ではなくAPI費用
   → 売上10倍でも人員は3倍以下

3. 【知識がコードに結晶化されている】
   ノウハウ・判断基準・業務プロセスがプロンプト・エージェント・コードとして蓄積
   → 個人の頭にある知識ではなく、組織の資産
```

### 判定チェックリスト

```
あなたの事業はAI Agent Companyか？

□ 1人あたり売上が業界平均の3倍以上（少数精鋭×AI高生産性）
□ 新しい仕事を受けるとき、まず「AIで自動化できるか」を考える
□ 業務プロセスが文書化→プロンプト化→エージェント化のパイプラインがある
□ 顧客への価値提供の速度が人間主体の競合の5倍以上
□ 夜中でもAIエージェントが稼働している
□ 売上が増えても比例して人件費が増えない費用構造

6つ全てYes → 真のAI Agent Company
4-5つ → AI Enabledカンパニー（移行中）
3つ以下 → 従来型（AI活用にとどまっている）
```

## 2. 組織設計原則

### 人員配置の原則

```
AI Agent Companyの役割分担:

AIエージェント（自動化対象）:
  - データ収集・整理・分析
  - 定型レポート・ドキュメント生成
  - コード生成・テスト実行
  - カスタマーサポートの初期対応
  - マーケティングコンテンツ生成
  - スケジューリング・タスク管理

人間（AIが代替できないもの）:
  - 最終意思決定・GoNoGo判断
  - 顧客・パートナーとの関係構築（信頼・感情）
  - 倫理的判断・価値観の設定
  - 予測不能な例外への対応（AIが失敗したとき）
  - 組織文化・価値観の体現
  - 新しい業務のプロンプト設計（AIを教える）

ハイブリッド（人+AI協働）:
  - 戦略立案（AIが情報収集・人が判断）
  - 提案書・営業（AIが素材・人が関係）
  - プロダクト設計（AIが実装・人がUX判断）
```

### 少数精鋭スケーリング原則

```
従来型スタートアップの成長曲線:
  売上10億円 → 従業員100-200名

AI Agent Companyの目標:
  売上10億円 → 従業員10-30名
  1人あたり売上: ¥3,000万〜¥1億

実現するための設計:
  1. 業務の80%をAI自動化してから採用する
  2. 採用は「AIを教えられる人」「AIが苦手な判断ができる人」に限定
  3. 人員増加より機能拡張（AIエージェント追加）でスケールする
  4. アウトソースより内製（外部への依存はコスト・品質管理リスク）
```

## 3. AIエージェント設計フレームワーク

### エージェント設計の5原則

```
原則1: 単一責任
  1つのエージェントは1つの役割に特化する
  ✅ Good: 「競合分析エージェント」
  ❌ Bad: 「なんでもやるエージェント」
  → 責任が明確なほど品質が上がり、デバッグが容易

原則2: 明示的な入出力仕様
  エージェントの入力・出力フォーマットを必ず定義する
  曖昧な入力 → 曖昧な出力 → 品質不安定
  「インプット: JSON形式の競合リスト、アウトプット: 差別化マトリクス」

原則3: 失敗ハンドリング
  AIエージェントは必ず失敗する。失敗時のフォールバックを設計する
  → エラー検出 → 人間にエスカレーション → 再試行ロジック

原則4: 人間監査ポイント
  完全自動化は危険。重要な意思決定前に必ず人間確認ステップを入れる
  High Stakes（顧客への送信・支払い処理・公開コンテンツ）は必須

原則5: 継続的改善ループ
  エージェントの出力品質をモニタリングし、プロンプトを継続改善する
  → ログ収集 → 品質評価 → プロンプト最適化 → 再デプロイ
```

### エージェントアーキテクチャパターン

```
パターン1: オーケストレーション型（ConsultingOSのアプローチ）
  orchestrator → [agent_A, agent_B, agent_C] → 統合
  用途: 複雑な多段階タスク

パターン2: パイプライン型
  agent_A → agent_B → agent_C（直列）
  用途: 情報が変換・加工されながら流れるワークフロー

パターン3: RAG型（Retrieval-Augmented Generation）
  user query → 検索 → 関連ドキュメント取得 → LLM生成
  用途: 社内知識ベース・FAQ・ドキュメント検索

パターン4: React/Tool Use型
  LLM → ツール選択 → ツール実行 → 結果評価 → 次アクション
  用途: 自律的なリサーチ・コード実行・Web操作

パターン5: マルチエージェント協議型（Agent Teams）
  複数エージェントが同じ問いに異なる視点で回答 → 議論 → 収束
  用途: 高品質な意思決定・反証によるバイアス除去
```

## 4. AI Native Product 設計思想

### プロダクトバリューの進化（佐藤裕介の知見を拡張）

```
AI Native Productの価値ピラミッド:

Level 4: 自律型（最高価値・到達目標）
  AIが目標を受け取り、計画→実行→評価まで自律的に行う
  人間は目標設定と最終承認のみ

Level 3: 協働型（現在の最前線）
  AIが95%の作業を行い、人間が判断・修正・承認
  ConsultingOSがここを目指す

Level 2: アシスト型（多くのSaaSの現在位置）
  人間が主体、AIがサジェスト・補完・自動化補助
  Copilot、文章補完、レコメンデーション

Level 1: 自動化型（RPA時代から継続）
  定型業務の自動化。AIではなくルールベース自動化も多い

→ 2年以内にLevel 2以下は競争優位がなくなる
→ Level 3以上で戦う設計を今から行う
```

### AI Native GTMの特徴

```
従来型SaaS GTM:
  認知 → 興味 → 体験（デモ/トライアル） → 購入 → 活用 → 継続

AI Native GTM（PLG進化型）:
  認知 → 即試行（30秒でAI成果体験） → 価値実感（WOW moment）
  → 習慣化 → 自然拡散（バイラル） → アップセル

WOW momentの設計:
  「AIを使い始めてから最初の感動ポイント」を明確定義・最短化する
  例: 「3分で競合分析レポートが完成する体験」
      「昨日までは1時間かかっていた作業が10秒で終わる体験」

AI Native ウイルス係数:
  従来: 「このプロダクト良いよ」口コミ
  AI Native: 「AIが作ったこの成果物を見て」アウトプット共有
  → アウトプットがマーケティング素材になる設計
```

## 5. AI Native Company の収益モデル

### 推奨収益モデル

```
モデル1: 価値ベース課金（最推奨）
  AIが生み出した価値の一部を課金
  例: 「節約した時間 × 人件費単価 × 30%」
  メリット: 顧客のROIが明確、価格交渉が「どれだけ節約するか」の議論になる
  デメリット: 価値測定の仕組みが必要

モデル2: 従量課金（スケールに適合）
  AIアクション数・API呼び出し数で課金
  例: レポート生成1件あたり¥500
  メリット: 顧客の使った分だけ請求（低参入障壁）
  デメリット: 使われると赤字になるラインの管理が必要

モデル3: サブスクリプション（安定収益）
  月額/年額固定
  メリット: 予測可能な収益、NRRの管理がしやすい
  デメリット: 実際の利用量とのミスマッチリスク

ハイブリッド推奨:
  基本サブスク（固定費カバー）+ 従量課金（スケール分）
  例: 月額¥30,000（基本機能） + ¥100/エージェント実行超過分
```

### AI Cost Management（原価管理）

```
AI Cost per Unit の管理:
  API費用（Claude/GPT）は変動費の主体
  売上成長に伴うAPI費用増加率を常にモニタリング

  コスト最適化の優先順位:
  1. プロンプト最適化（トークン削減 → 即座にコスト削減）
  2. キャッシュ活用（同じ入力への繰り返しAPI呼び出しをキャッシュ）
  3. モデル選択（全てをOpusに頼らず、タスクに応じてモデルを使い分け）
  4. バッチ処理（リアルタイム不要なタスクはバッチAPIで50%コスト削減）

モデル選択指針:
  Claude Opus 4.6  → 最高品質が必要（提案書・重要判断・複雑なコード）
  Claude Sonnet 4.6 → バランス型（日常的なタスクの主力）
  Claude Haiku 4.5 → 高速・低コスト（分類・要約・定型処理）

粗利率管理:
  目標: AI Agent Companyの粗利率 ≥ 70%
  API費用が売上の15%以上になったら最適化アラート
  API費用が売上の25%を超えたら緊急対応（モデル変更・プロンプト最適化）
```

## 7. よくある失敗パターンと対策

### AI Native移行の落とし穴

```
失敗1: AIに全部任せすぎる（品質崩壊）
  症状: AIアウトプットをノーチェックでクライアントに送る
  対策: 重要度に応じた人間確認レイヤーを設計する
        High: 必ず人間確認、Medium: サンプリング確認、Low: 全自動

失敗2: 人間の仕事をAIに置き換えるだけ（真の自動化になっていない）
  症状: 人間の作業をAIが代わりにやるだけで、構造が変わっていない
  対策: まずビジネスプロセスを再設計してからAIを入れる
        「このプロセス自体が必要か？」から問い直す

失敗3: AIの失敗を想定していない（脆弱な自動化）
  症状: AIが間違った出力をしたとき、誰も気づかない
  対策: 失敗検出ロジック + エスカレーションフロー + ログ監査を必ず実装

失敗4: 過剰な人員削減で組織能力が落ちる
  症状: AIで自動化したと同時に人員を削減し、例外処理ができなくなる
  対策: 自動化を先行、人員調整は半年後（自動化の安定確認後）

失敗5: AIの進化に設計が追いつかない
  症状: 6ヶ月前に作ったプロンプトが現在のモデルに最適化されていない
  対策: 四半期ごとのプロンプト・エージェント設計の見直しサイクルを持つ
```
